# 파이널 프로젝트(6주 캡스톤) 심화 레퍼런스

## I. 큰 그림 — 의뢰인 6주 환경의 좌표

### 정합도 스펙트럼 (의뢰인 환경 = 6주 + GitHub SSOT + AC + 24h PR SLA + 매주 데모 + Week 6 Freeze)

```
[정합도 최상] ─────────────────────────────────── [참고 가치 있음]

🏆 Turing Mod 4 (6주, GitHub Projects, AC 체크리스트, PR 강사 review 2회)
🏆 부스트캠프 안드로이드 멤버십 (6주, GitHub Projects SSOT, 주간 KPT)
🏆 Apple Developer Academy Foundation (4~6주, CBL 3-pillar, 모바일)

   ├─ SSAFY 자율 프로젝트 (6주, Jira+GitLab+Notion, 1주 스프린트)
   ├─ Codesmith OSP (5주, staggered, 4명 팀)
   ├─ Google Solution Challenge (2.5개월, 2분 데모 영상)

   ├─ 우테코 안드로이드 (Lv3+4, 데모데이 부스+투표)

   ├─ Ironhack Final (2주, 5분+1분 2종 발표)

   │
   └─ 42 ft_transcendence (3개월), App Academy, GA SEI 등
```

### 결정적 발견 4개

1. **🏆 Turing School Module 4**가 의뢰인 환경과 **거의 1:1 대응** — 6주 + GitHub Projects SSOT + AC 체크리스트 + 카드 4종(MVP/chore/bug/spike) + Week 1 "no code" 룰 + 강사 PR review 2회 의무 + 10분 최종 데모
2. **의뢰인 환경의 PR 24h SLA 수치 명시**는 해외·국내 사례에서 **단 하나도 찾지 못함** → 의뢰인 운영의 차별점
3. **멋쟁이사자처럼 안드로이드 부트캠프 1~5기**의 파이널 프로젝트 공개 후기는 검색 결과 **0건** → **의뢰인 6기 운영이 사실상 공개 베스트 프랙티스의 1차 사례**가 될 가능성
4. **6주 + GitHub SSOT + Week 6 Freeze + 매주 데모**까지 모두 명시한 곳은 의뢰인 환경 외에 없음 → 단일 reference보다 **더 구조화된 형태**

## III. 가장 정합도 높은 3개 사례 심층

### 🥇 Turing School Module 4 Capstone (미국) — **거의 1:1 대응**

**기간**: 정확히 **6주**. Week 1은 코딩 금지·"Inception" 기획만. 이후 2~3일 단위 sprint.

**팀**: 크로스 프로그램(FE+BE) 혼합 팀. 강사가 PM 역할 겸임.

**워크플로**:
- **단일 GitHub Projects 보드 = SSOT** (의뢰인 환경과 동일)
- Slack 채널, CircleCI 초기 설정 필수
- 이슈에 description + **acceptance criteria 체크리스트** (의뢰인 AC 3~7개와 정합)
- "1일 안에 끝낼 만한 크기, 더 크면 쪼개기" 규칙
- 보드 카드 종류 4가지: **MVP user stories / chores / bugs / spikes**

**Definition of Done**: 의뢰인 환경과 동일한 **PR-merge 기반** + **강사 PR review 2회 의무**(FE 1회 + BE 1회)

**사전 문서**: Wireframes + Schema designs + JSON contracts 3종을 Week 1 inception 산출물로 검토 → 의뢰인의 Product Brief → User Flow와 거의 동등

**데일리/위클리**:
- Daily stand-up (어제/오늘/blocker 3분할)
- Weekly retros + backlog refinement 캘린더 고정
- 강사 주재 check-in

**데모**: 10분 최종 데모(End of Project Demos), production 라이브 데모 권장

**평가**: rubric 기반 (MVP plan / board hygiene / communication 등)

**의뢰인 시사점**:
- Turing의 "PR review with instructor 2회 의무"는 의뢰인 24h SLA보다 강제력이 강함 → 보조강사 PR 1차 피드백 책임 구조에 그대로 차용 가능
- "Week 1 = no code" 원칙은 의뢰인이 이미 운영 중이지만 명시화할 근거
- 카드 4종(MVP/chore/bug/spike) 분류는 의뢰인 라벨 체계 보강 후보

**출처**:

- https://mod4.turing.edu/projects/capstone/capstone_rubric.html

### 🥈 네이버 부스트캠프 안드로이드 멤버십 (국내) — **의뢰인 환경의 원형**

**전체 일정**: 멤버십 22주. 챌린지 4주 → 학습 스프린트 8주 → **그룹 프로젝트 6주**

**팀**: 표준 6명 (Android 2 + Frontend 2 + Backend 2). 안드로이드 only 팀(2025년 and01)은 4명.

**워크플로 도구**: **GitHub Projects + GitHub Actions + Discord 웹훅**
- 외부 도구(Jira 등) 권한이 차단되어 GitHub Projects를 SSOT로 강제 사용
- PR self-assign · label · Project 자동화
- Discord에 GitHub 웹훅 연결

**이슈/PR 규칙** (7기 회고):
- **백로그 이름 = PR명** (그라운드룰)
- dev 브랜치 머지는 **Squash merge** → 커밋 이력이 곧 백로그
- dev/main 머지 시 자동 배포 (CI/CD)
- Issue 템플릿 / PR 템플릿 / 커밋 템플릿 직접 제작

**스프린트**: 1주 단위, 금요일 **KPT 회고**가 정기 의례. 다음 주부터 적용.

**데모**: 매주 데모 발표 + 마지막 네트워킹 데이 전체 발표

**멘토**: 851명 현업 멘토 풀, 마스터클래스 별도, 코드 리뷰어 별도 모집

**번아웃 패턴** (9기 후기): "어떤 사람은 저녁 7시 마감, 다른 사람은 밤새 작업해도 못 끝냄… 회의 때마다 현타." 마지막 주 안정화는 개별적.

**공개 산출물**: 기수별 GitHub Org 공개 (`boostcampwm2021` ~ `boostcampwm2025`). 9기 80개 repo, 2025년 `and01-boostcamp` = "Memorip" (Kotlin + Jetpack Compose + Spring Boot, 4명, dev 브랜치 1,813 커밋)

**의뢰인 시사점**:
- 의뢰인의 "GitHub Issue SSOT + Notion 설명용" 모델과 정확히 동일
- "백로그명 = PR명" + "Squash merge로 백로그 이력 보존" 룰은 의뢰인 환경에 즉시 차용 가능
- 주간 KPT 금요일은 의뢰인 신호등 회고의 상위 호환 패턴

- **데모데이**: 5기 2023.10.20 우아한테크코스 잠실 캠퍼스, **부스 시연**. 우아한테크캠프 7기는 **방문자 투표제**로 1위 결정
- **워크플로**: GitHub PR 중심. 팀 자체 그라운드룰 (브랜치 전략, 컨벤션, 코드 리뷰 절차)

### 3. SSAFY 자율 프로젝트 (안드로이드)

- **기간**: **6주** (9기: 2023.10.09~11.17). 8기 자율은 첫 3주에 빠른 MVP → 1차 배포 → 사용자 분석(GA, HotJar)으로 2회 추가 배포
- **팀**: 6인 표준 (안드로이드 트랙 시 갤럭시 스마트폰 1대 지급)
- **워크플로**: **Jira + GitLab + Notion + Figma** 분할 운영
  - Jira: 이슈/스프린트
  - GitLab: 사내 Git, PR
  - Notion: 회의록·요구사항·일정·스크럼·발표 정리
- **이슈/PR 규칙**: Git/Jira/FE/BE 컨벤션 별도 문서화. 커밋 type: `init, feat, fix, refactor, docs, test, chore`
- **스프린트**: 1주 단위 + 매일 5분 **데일리 스크럼** (어제/오늘/장애요인). 칸반 보드
- **데모**: 지역 발표(반 단위) → **결선 발표(전국)** + 외부 심사 수상
- **공개 산출물**: heart-ing(8기 최우수), MOASS(10기 최우수), MYINI, dallim, pocket-fridge 등
- 출처: https://www.ssafy.com/swp/rps/req/Leaflet_202310.pdf , https://github.com/wjdwn03/heart-ing , https://github.com/RosaDamascena/MOASS

### 4. 우아한테크캠프 7기 (10주, 백엔드 중심·참고용)

- 출처: https://techblog.woowahan.com/19370/

## VI. 학술·실무 문헌 (10개 영역)

### 1. SE 캡스톤 Pedagogy (ACM/IEEE)

#### 1.1 ★ SE2014 Curriculum Guidelines (ACM/IEEE-CS)
- **원전**: Ardis et al. (2015). *Software Engineering 2014: Curriculum Guidelines for Undergraduate Degree Programs in Software Engineering*. DOI 10.1145/2594168
- 캡스톤은 SE 학사의 "integrative experience" — (a) 실제 클라이언트, (b) 팀 운영(Forming~Performing), (c) **프로세스 가시화(공정 모델 명시)**, (d) 산출물뿐 아니라 "프로세스 학습"을 평가
- **의뢰인 직결**: Product Brief → AC → PR 흐름 = SE2014의 "process visibility" 그대로
- 자료: https://www.acm.org/binaries/content/assets/education/se2014.pdf

#### 1.2 ★ 캡스톤 SLR (IST 2023) — 127편 분석 — **최우선 entry point**
- **원전**: *Information and Software Technology* 158 (2023), Article 107191. DOI 10.1016/j.infsof.2023.107191. arXiv 2301.03554
- **핵심 결론**:
  - 캡스톤 중앙값은 1학기. **6주 미만은 소수**지만 부트캠프 영역에서 증가
  - 팀 규모: **4–5명이 mode** (의뢰인 3–4명과 합치)
  - 산출물: "proof-of-concept" 권고 (의뢰인 "2~5분 데모"와 합치)
  - **짧은 캡스톤의 가장 빈번한 실패**: 요구사항 폭주 후 **후반부 통합 실패** = SE2014의 process visibility 부재
- **의뢰인 직결**: GitHub Issue SSOT + 24h SLA = process visibility의 운영적 답. Week 6 Freeze = 후반부 통합 실패의 직접 방어책
- 자료: https://arxiv.org/abs/2301.03554

### 2. Industry-Sponsored / PBL

#### 2.1 Sponsor Best Practices (Colorado Boulder)
- 스폰서는 **Technical Lead 1인** 지정, 최소 주 1시간 commit
- 1차 단계에서 **scope written agreement** — "good-to-have" 위주, customer critical path 금지
- 우수 vs 실패 프로젝트의 결정적 차이 = **스폰서 관여 빈도**
- **의뢰인 직결**: 보조강사가 "도메인 expert/대리 PO" 역할이면 주 1시간 commit 보장이 임계점. AM/PM 10분 미팅은 미달이므로 **AC 합의·블로커 해소를 위한 30분 동기 sync 1회/주 추가 권장**
- 자료: https://www.colorado.edu/cs/engage-us/sponsor-computer-science-capstone-project

#### 2.2 PBLWorks Gold Standard (Buck Institute)
- **7요소**: Challenging Problem / Sustained Inquiry / **Authenticity** / Student Voice & Choice / Reflection / **Critique & Revision** / **Public Product**
- **Public Product**: 외부 청중 노출이 학생 quality bar를 끌어올림
- **Critique & Revision**: "right/wrong" → "continuously improving" 마인드셋
- **의뢰인 직결**: 매주 데모 = "interim public product", Week 6 데모 = "final public product". 24h PR SLA + AC = critique & revision의 형식화
- 자료: https://www.pblworks.org/what-is-pbl/gold-standard-project-design

#### 2.3 ★ Sprint Zero / Inception Deck (Rasmusson 2010)
- **원전**: Jonathan Rasmusson, *The Agile Samurai* (2010)
- **10 질문**: Why are we here / Elevator Pitch / Design the Product Box / **NOT List** / Meet Your Neighbors / Show the Solution / What Keeps Us Up at Night / Size It Up / Trade-offs (시간/품질/범위/예산) / What's It Going to Cost
- **의뢰인 직결**: Week 0 Brief 단계에서 **NOT List**를 강제하면 학생들의 가장 빈번한 실패(scope creep)를 사전 차단. Trade-off slider는 Week 5 freeze 결정의 근거
- 자료: https://rasmusson.files.wordpress.com/2008/01/rasmusson-agileinceptiondeckbootcamp.pdf

### 3. Agile in Education (Scrum at Capstone)

#### 3.1 ★ Mahnic (2012, IEEE TE) — 캡스톤+Scrum 표준 reference
- **원전**: Mahnic, V. (2012). "A Capstone Course on Agile Software Development Using Scrum". *IEEE Transactions on Education* 55(1):99–106. DOI 10.1109/TE.2011.2142311
- 7개 학기 데이터. 학생 = Scrum Team, 강사 = **Product Owner 대리**
- 학생 자신감과 추정 정확도가 **sprint 4 이후 통계적으로 유의미 향상**. **Sprint 1–2는 추정 underrun (40~60% 오차)**, Sprint 3 이후 수렴
- **의뢰인 직결**:
  - 보조강사 = PO 대리 모델은 Mahnic이 검증한 패턴
  - 6주에서는 sprint 4=Week 4 시점에 추정 정확도 잡힘 → **Week 5–6 freeze는 합리적**
  - **Week 1–2 추정 오차는 정상**임을 학생에게 미리 고지하면 burn 완화
- 자료: https://peerassessment.com/wp-content/uploads/2021/08/A-Capstone-Course-on-Agile-Software-DevelopmentUsing-Scrum-Mahnic-2012.pdf

#### 3.2 Rico & Sayani — PBL+Agile 캡스톤
- PO 역할 외부 고객 → 내부 교수/TA 위임 시 **AC 명세 품질이 성공 변수**
- AC 모호 시 학생은 "**fake done**" 경향
- **의뢰인 직결**: "Done = PR Merge + AC 충족"의 두 번째 절반(AC 충족)이 **학생 자체평가가 아니라 PO 대리(보조강사) 확인**이어야 fake done 차단

#### 3.3 Scrum Master 역할 (Scrum.org)
- **6 stances**: Servant Leader / Facilitator / Coach / Mentor / Manager / Impediment Remover
- **의뢰인 직결**: 보조강사 = **"Impediment Remover + Coach"** 우선. 24h PR SLA 위반 시 impediment escalate, 회고 시 coach 모드
- 자료: https://www.scrum.org/resources/blog/28-characteristics-great-scrum-master

#### 3.4 Acceptance Criteria 교육법
- **3 Cs** (Ron Jeffries): Card / Conversation / **Confirmation(=AC)**
- **Given-When-Then** (Gherkin) 양식
- **의뢰인 직결**: GitHub Issue 템플릿에 Given-When-Then을 필수 필드로 두면 AC 누락 구조적 차단
- 자료: https://www.atlassian.com/agile/project-management/user-stories

### 4. Demo Day / Pitch / Showcase

#### 4.1 ★ Y Combinator Demo Day 공식 가이드 — **2분 데모 절대 reference**
- **원전**: YC Library, "A Guide to Demo Day Presentations"

- **Rule of Three** / **Repeat one-liner at slides 1, 9, 14**
- **타임라인**: T-14 script / T-10 record yourself & delete 30% of words / T-7 pacing / T-5 timed rehearsal / T-3 group feedback / T-1 polish / T-0 30min early
- **의뢰인 직결**: 6주 매주 데모 2~5분 = YC 형식과 거의 정확 매칭. T-3 그룹 피드백 = Week 5 리허설, T-1 = Week 6 freeze 직전 polish

#### 4.2 Capstone Rubric 공개 사례

  - **Backup mp4**: 라이브 실패 시 30초 내 전환

### 5. MVP Scoping & Mid-project Pivot

#### 5.1 ★ Walking Skeleton (Cockburn 2004) — **Week 1 운영 reference**
- **원전**: Cockburn, A. *Crystal Clear: A Human-Powered Methodology for Small Teams*, Addison-Wesley, 2004
- **정의**: "A tiny implementation of the system that performs a small end-to-end function. It need not use the final architecture, but it should link together the main architectural components."
- **유사 개념**: Pragmatic Programmer "Tracer Bullet", Gojko Adzic "Walking Skeleton on Crutches"
- **의뢰인 직결**: Week 1 안에 **(Android UI → ViewModel → Repository → Network → 가짜 응답) end-to-end 한 줄 동작** 강제. 이후 5주는 살을 붙이는 게 아니라 **그 줄을 따라 깊이를 늘리는 작업**. 이 패턴이 짧은 캡스톤의 가장 큰 통합 리스크 제거
- 자료: https://wiki.c2.com/?WalkingSkeleton

#### 5.2 MoSCoW with "Won't-Have" 강조
- 학생팀에서 가장 강력한 anti-creep 도구는 Must/Should/Could가 아니라 **Won't-Have 명시**
- "당신 아이디어는 잊혀지지 않았으나 이번 주기엔 안 함"이라는 명시적 발언이 신뢰를 만든다
- **의뢰인 직결**: Week 0 GitHub Issue에 `wont-have` 라벨/별도 보드 컬럼. PO 대리(보조강사)가 매주 한 줄 새 won't-have 결정 요구

#### 5.3 PMI scope creep 통계
- 60명 PM 서베이: **92%의 프로젝트 실패가 scope creep 미관리와 연결**
- 결정적 변수: **formal change control process** 존재 여부
- **의뢰인 직결**: Issue SSOT = formal change control. Week 3 시점 "scope change request" template 만들고, AC 새 요구는 반드시 기존 must를 won't로 강등하는 거래 형식 강제
- 자료: https://www.pmi.org/learning/library/top-five-causes-scope-creep-6675

### 6. Capstone Burnout & Team Dynamics

#### 6.1 ★ Tuckman Forming-Storming-Norming-Performing-Adjourning
- **원전**: Tuckman 1965 / Tuckman & Jensen 1977
- **6주 캡스톤 매핑**:
  - Week 1 **Forming**
  - Week 2 **Storming** ← conflict 정상 신호
  - Week 3 **Norming**
  - Week 4–5 **Performing**
  - Week 6 **Adjourning**
- **개입 지점**: Storming(Week 2)에서 conflict가 normal임을 명시 공지 + 역할 명확화 + psychological safety
- **의뢰인 직결**: **R-burn-1 발현(2026-05-14 AM, Week 4 부근)** = Performing 전환 압력과 정확 매칭. Tuckman 관점에선 이때 leader가 task focus 줄이고 **process facilitation 늘려야** 함. 보조강사 모드를 Coach → Impediment Remover로 명시 전환할 시점
- 자료: https://hr.mit.edu/learning-topics/teams/articles/stages-development

#### 6.2 부트캠프 캡스톤 stress 강도 (vs 학사)
- IST 2023 SLR: 부트캠프 캡스톤은 (a) 외부 직장 연결 보상 압력 + (b) 짧은 기간 → **stress intensity가 학사보다 1.5~2배**
- sustained duration 짧음 → **chronic burnout보다 acute burnout 패턴**
- **의뢰인 직결**: 일일 점검(AM/PM)은 acute burnout 조기 감지에 최적. R-attend 같은 약한 신호 추적 효과적

#### 7.2 ★★ Hundhausen et al. (2023, ACM TOCE) — **의뢰인 환경 최강 직접 reference**
- **원전**: Hundhausen, C., Conrad, P., Adesope, O., Tariq, A. (2023). "Combining GitHub, Chat, and Peer Evaluation Data to Assess Individual Contributions". *ACM TOCE* 23(3), Article 33. DOI 10.1145/3593592

- **의뢰인 직결**: Issue SSOT + PR Merge로 GitHub 메트릭은 이미 자동 수집됨. 여기에 **간단한 동료 평가 1회(Week 3) + 1회(Week 6) 추가**하면 학술 검증된 fairness 수준 도달. 기여 불균등(38/50%)은 정섭 결석 사례와 통계 합치 → **보조강사 사전 인지 가능**
- 자료: https://pconrad.github.io/files/paper034.pdf (무료)

## VII. 의뢰인 환경 즉시 적용 가능한 처방 6개 (학술+해외+국내 합성)

### 1. Week 0: Rasmusson Inception Deck 핵심 5질문
- **출처**: Rasmusson (2010) 10 질문 중 5개 추림
- **방법**: 1시간 워크숍에서 (a) Why are we here, (b) Elevator Pitch, (c) **NOT List** (Won't-Have), (d) Trade-off slider, (e) What Keeps Us Up at Night 만 응답

- **출처**: Cockburn (2004) + Turing Mod 4 "Week 1 no code"

- **미달 팀**: 보조강사가 직접 page-through (Mahnic 2012: Sprint 1–2 추정 underrun 정상 → push 모드 정당화)
- **효과**: 후반부 통합 실패 (IST 2023 SLR 최빈 실패) 직접 방어

### 3. Week 3 & 6: 간이 CATME + RepoSense fairness check
- **출처**: Hundhausen 2023 (TOCE) — 객관(GitHub) + 주관(CATME) 결합
- **방법**:
  - Google Form 5문항 (CATME 5축 간이판)
  - `npx reposense` 로 GitHub 기여 시각화 CLI 한 줄
  - 두 결과 결합 → 기여 불균등 38/50% 사례 사전 감지

- **방법**:
  - Tagline → Problem → Solution → How it works → Traction → Tech → Ask
  - **Rule of Three**: 핵심 one-liner를 slide 1, 9, 14에서 반복
  - **scrcpy USB tether 미러링** + **30초 mp4 백업** 항상 준비

### 5. Tuckman 기반 보조강사 모드 전환
- **출처**: Tuckman 1965 + Scrum.org 6 stances
- **방법**:
  - **Week 1 Forming**: 보조강사 = Facilitator (역할 명확화 가이드)
  - **Week 2 Storming**: 보조강사 = Coach (conflict가 normal임을 공지)
  - **Week 3 Norming**: 보조강사 = Mentor (페어 매칭, 그라운드룰 확정)
  - **Week 4–5 Performing**: 보조강사 = **Impediment Remover** (블로커 제거 우선)
  - **Week 6 Adjourning**: 보조강사 = Coach (회고·다음 단계 가이드)
- **R-burn-1 발현 시점**: Storming→Performing 전이 압력 = Week 4 부근, 이때 mode 전환 명시화

### 6. 부스트캠프식 GitHub 워크플로 차용
- **출처**: 부스트캠프 7기 회고
- **방법**:
  - **백로그명 = PR명** 룰 (이슈 ↔ PR 매핑 일관성)
  - **dev 머지 = Squash merge** (커밋 이력 = 백로그)
  - Issue / PR / 커밋 템플릿 의뢰인 환경 v2 양식과 통합
- **효과**: 의뢰인 가이드의 "PR 본문에 Closes #N 명시" 룰과 정확 합치

### 3. 멋사 1~5기 파이널 공개 후기 0건 → 의뢰인 6기 운영이 1차 베스트 프랙티스

### 5. R-burn-1 발현 시점 = Tuckman Storming→Performing 전이
2026-05-14 AM의 burn 신호는 우연이 아니라 6주 캡스톤의 정상 패턴. Mahnic 2012 sprint 4 (=Week 4) 시점 추정 정확도 수렴과도 합치. **보조강사 stance를 Coach → Impediment Remover로 전환할 시점**.

### 6. AC 모호 시 "fake done" 위험 — 보조강사 PO 대리 검증 필수
Rico & Sayani 연구: PO 역할이 외부 고객 → 내부 TA 위임 시 AC 명세 품질이 결정 변수. **Done = PR Merge + AC 충족**의 "AC 충족" 판정은 학생 자체평가가 아닌 **보조강사 확인** 필수.

### 7. 모바일 캡스톤 라이브 데모 실패율 3× — 백업 mp4 표준화
ASEE 보고. 안드로이드 라이브 시연은 USB/WiFi 끊김 + 디바이스 호환성 이슈로 데스크톱보다 3배 실패. **scrcpy + 30초 mp4 백업**이 학술 권고 표준.

### 8. 의뢰인 v2 양식 + carry-over + 신호등 회고가 Hundhausen 2023의 한국형 운영체화
객관(GitHub Issue SSOT) + 주관(신호등 회고·R-항목) 결합이 학술 검증된 fairness 모델. 의뢰인 운영이 이론적 토대 있음을 확보.

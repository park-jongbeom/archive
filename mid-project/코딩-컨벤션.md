# 코딩 컨벤션 for AOS 6기 (MVVM + Feature-based)

이 문서는 코드 구조·작성 습관(컨벤션)을 정리한다. Firestore 같은 설계는 설계 가이드를 먼저 보고, 여기서는 구현 단계 품질에 맞춘다.

---

## 1. 적용 범위

- Kotlin
- Jetpack Compose
- 아키텍처: `MVVM + Feature-based`
- `feature/...` 기본: `ui`(Compose), `viewmodel`(화면별), `data`(Repository + 그 기능 전용 모델). DataSource는 선택. ([개발 가이드](./개발-가이드.md) §1)

---

## 2. 기본 원칙(한 문단 요약)

- View(UI)는 UI 렌더링과 이벤트 위임만 담당한다.
- ViewModel은 `UiState`를 만들고 사용자 이벤트를 받아 Repository 호출/매핑을 조정한다.
- 도메인/데이터 변환은 ViewModel에서 흩어지지 않게 Repository 또는 Mapper 계층에 모은다.
- 상태는 불변(immutable)으로 다루고, UI는 상태를 “읽기”만 한다.

---

## 3. Feature-based 폴더 규칙

- 기능은 `feature/<기능명>/` 아래에서 스코프를 유지한다. 아래 트리는 패키지 예시이며, Gradle feature 모듈로 나눠도 `ui` / `viewmodel` / `data` 구분은 같게 가져가면 된다.
- 공통은 `core/`(또는 `shared/`)로 둔다.
- feature 간 직접 의존은 최소화한다.
  - 공유 모델/인터페이스, 공통 이벤트/내비게이션 계약 같은 형태로만 연결한다.

### 예시(개념)

```text
com.example.app/
├── core/
│   ├── ui/
│   ├── network/
│   └── util/
├── feature/
│   ├── login/
│   │   ├── ui/
│   │   ├── viewmodel/
│   │   └── data/
│   └── profile/
│       ├── ui/
│       ├── viewmodel/
│       └── data/
└── MainActivity.kt
```

폴더별 기본: `ui` = 화면(View), `viewmodel` = 화면 단위 ViewModel, `data` = Repository + 모델(DTO 등), DataSource는 필요할 때만.

---

## 4. MVVM 규칙

### View(Composable)

- 가능한 한 “Stateless”에 가깝게 작성한다.
- `Flow`/`StateFlow`는 `collectAsState()` 등으로 관찰해 UI를 그린다. Lifecycle에 맞추려면 `lifecycle-runtime-compose`의 `collectAsStateWithLifecycle()` 도입을 검토한다(프로젝트·버전에 맞게).
- UI의 모든 사용자 이벤트는 `viewModel.onXxx()`처럼 ViewModel 메서드로 전달한다.
- View에서 네트워크/DB/토큰 검증 같은 로직을 두지 않는다.

### ViewModel

- `UiState`를 `StateFlow`(또는 `LiveData`)로 제공한다.
- 이벤트 처리 메서드는 보통 `onXxx()` 네이밍을 사용한다.
- `loading / success / error` 패턴을 유지한다.
  - 예: `UiState.Loading`, `UiState.Content`, `UiState.Error`
- ViewModel에서 Repository 호출과 결과 매핑을 처리한다.
  - 변환이 커지면 Mapper/UseCase로 분리해 ViewModel이 과도해지지 않게 한다.
- Hilt 등 DI를 쓰면 Repository·UseCase는 `@Inject` 생성자 등으로 주입하고, Compose는 `hiltViewModel()` 같은 팀 규칙을 따른다(패키지 예시는 [개발 가이드](./개발-가이드.md) §1).

### 테마·로딩·에러 (앱 초기에 맞추기)

- 테마는 `core/design` 등 한곳에서만 쓰고, 화면에서 색·타이포를 하드코딩하지 않는다(예외는 최소).
- 로딩은 `UiState`에 `isLoading` 또는 `Loading`/`Content`/`Error` 같은 구분을 두고, 공통 로딩 컴포저블(또는 전역 오버레이)을 팀에서 하나 정한다.
- 에러는 `Throwable.message`를 그대로 노출하기보다, UI용 타입(문자열 리소스 ID + fallback 등)으로 바꿔서 보여 준다. 네트워크 실패·권한 거부·입력 검증 실패를 구분할지는 팀 범위에 맞게 정한다.
- 재시도·닫기 같은 공통 동작이 있으면 동일 패턴으로 맞춘다.

---

## 5. Kotlin 네이밍/형식 규칙

### 패키지

- 소문자, 점(.)으로 구분: `com.example.app.feature.login`

### 클래스/데이터 클래스

- 파스칼케이스: `LoginUiState`, `UserProfile`

### 함수/프로퍼티

- 카멜케이스: `loadProfile()`, `onLoginClicked()`

### boolean

- `is/has/can` 접두어: `isLoggedIn`, `hasMore`, `canRetry`

### 상수

- 팀 규칙에 맞춘다(예: `DEFAULT_PAGE_SIZE`).

---

## 6. Compose 컨벤션

- Composable은 가능한 한 “읽기 + 이벤트 위임”에 집중한다.
- `remember`/`derivedStateOf`는 성능 최적화가 필요할 때만 사용한다.
- 부수효과는 `LaunchedEffect` 등으로 제한적으로 처리한다.
  - 반복 실행이 문제될 수 있는 로직은 key를 명확히 지정한다.
- 리스트 화면은 `LazyColumn/LazyRow` 사용을 우선하고, 키(`key`)를 설정한다(가능한 경우).

---

## 7. Coroutines / Flow 컨벤션

- Repository의 외부 작업은 `suspend` 또는 `Flow`로 노출한다.
- ViewModel에서는 `viewModelScope`로 코루틴을 실행한다.
- UI에서는 “상태 수집(collect)” 외의 스레딩/디스패치 제어를 하지 않는다.

---

## 8. 네트워크/데이터 처리 규칙(원칙 중심)

- DTO(외부 포맷)와 내부 모델/Ui 모델을 분리한다.
- 변환 책임은 Mapper/Repository에 둔다.
- 에러는 문자열로 흘리기보다 UI용 타입으로 구조화한다.
  - 예: `UiError(messageResId, fallbackMessage)`
- Repository에서 IOException·FirebaseException 등을 잡아 도메인/UI 계층에 맞는 결과로 바꿔도 되고, ViewModel에서 매핑해도 된다. 어느 쪽이든 한 화면·한 기능 안에서만 흩어지지 않게 한다.
- Firestore/DB의 타입 변환은 “한 곳”에서 끝낸다.

---

## 9. 테스트 가이드(범위 설정)

- Repository: 매핑/예외 케이스 중심
- ViewModel: `UiState` 변화 중심
- UI(Compose): 핵심 흐름 위주로(과도한 UI 테스트는 지양)

---

## 10. PR 리뷰 체크리스트(팀 합의용)

- UI 로직이 ViewModel/Repository로 옮겨져 있는가?
- ViewModel의 `UiState` / 이벤트 흐름이 규칙에 맞는가?
- feature 폴더 경계가 유지되는가?
- 에러/로딩 처리가 누락되지 않았는가? (팀이 정한 공통 패턴과 맞는지)
- 민감정보(키/토큰)가 커밋/코드에 들어가지 않았는가?

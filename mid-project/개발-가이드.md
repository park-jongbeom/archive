## 1. 코딩·아키텍처 합의

여기는 회의에서 숫자 맞춰 두는 수준의 요약이다. Compose·Coroutine·리뷰 때 볼 세부 규칙은 [코딩 컨벤션](./코딩-컨벤션.md)에 몰아뒀다. 두 문서에 같은 내용이 있으면 한쪽 고칠 때 다른 쪽도 같이 본다.

| 항목 | 합의 내용 |
|------|-----------|
| 언어 | Kotlin |
| UI | Jetpack Compose(팀 합의 기준) |
| 아키텍처 | MVVM + Feature-based |
| MVVM | View는 UI, 상태·이벤트는 ViewModel, 데이터는 Repository(등)로 분리 |
| Feature-based | 패키지/모듈을 기능 단위로 나누고, 공통은 `core` 등으로 추출 |

`feature/...` 하위 폴더 기본

| 폴더 | 기본으로 두는 것 |
|------|------------------|
| `ui/` | Compose 화면·컴포저블(View). 화면이 많으면 하위 패키지로만 나눈다. |
| `viewmodel/` | 화면(또는 좁은 플로우)당 ViewModel. 기능 화면이면 1:1 권장. |
| `data/` | Repository(ViewModel이 부르는 창구) + 이 feature 전용 모델(DTO·Firestore 매핑 등). Firestore·API는 Repository에서 직접 호출해도 된다. |
| `data/` (선택) | DataSource — 목/실출처를 클래스로 나누거나 출처가 여러 개일 때만. 없으면 Repository만으로 충분. |

기본 조합: `ui` + `viewmodel` + `data`(Repository + 모델). DataSource만 선택.

기능 화면과 ViewModel (권장)

- 화면 하나(또는 Nav destination 하나)당 ViewModel 하나(1:1)를 기본으로 두는 편이 읽기 쉽다. 예: 목록 ↔ `BoardListViewModel`, 상세 ↔ `BoardDetailViewModel`, 작성 ↔ `BoardEditorViewModel`.
- Navigation의 `viewModel()` / `hiltViewModel()` 스코프와도 맞물리기 좋다.
- 예외: 가입·결제처럼 한 플로우로 묶인 여러 화면은 Nav 그래프 상위에서 ViewModel을 공유할 수 있다. 다만 “화면 하나짜리 거대 ViewModel”은 피하고, 플로우 범위가 좁을 때만 공유하는 쪽을 권장한다.

세부 패키지 구조·네이밍은 팀 내 컨벤션 문서가 있으면 그쪽을 우선한다.

코딩 세부 규칙은 [코딩 컨벤션](./코딩-컨벤션.md)을 따른다.

패키지 구조 예시(개념)

팀마다 다르지만, 기능 단위로 묶을 때 아래와 비슷한 형태가 흔하다.

```text
com.example.app/
├── core/                 # 공통 UI·유틸 등 (아래 기본 예시)
├── feature/
│   ├── login/
│   ├── home/
│   └── profile/
└── MainActivity.kt       # 진입·네비게이션 등
```

`core` 예시 (팀에서 쓰는 것만 골라 씀)

여러 feature가 공유하는 코드만 둔다. 당장 없으면 패키지를 만들지 않아도 된다.

```text
core/
├── design/               # Theme, Color, Typography — Material·앱 전역 스타일
├── ui/                   # 공통 Compose 컴포넌트(버튼·텍스트필드·로딩)·레이아웃
├── util/                 # 확장 함수·검증·날짜·문자열 포맷 등
├── navigation/           # 공통 라우트 상수·타입·확장 (선택)
├── network/              # OkHttp/Retrofit 인스턴스·인터셉터 등 (REST 쓸 때만)
├── di/                   # Hilt @Module — 싱글톤·공통 바인딩 (해당 시)
└── model/                # 여러 feature가 같이 쓰는 얇은 타입·에러 래퍼 등 (최소만)
```

- Firebase만 쓰면 `network`는 생략해도 된다.  
- `model`은 비대해지기 쉬우니 feature 전용 모델은 각 `feature/.../data`에 두고, 정말 공통인 것만 올린다.

앱 초기 공통화 (권장)

스프린트 초반·골격(F2) 단계에서 팀이 한번 맞춰 두면 이후 화면이 덜 엇갈린다. 과제 기간이 짧으면 아래만 최소로 정해도 된다.

| 영역 | 보통 두는 위치 | 팀에서 정할 것 |
|------|----------------|----------------|
| 테마 | `core/design` | Color·Typography·다크 모드 여부, 간격 단위(예: 4dp 그리드) |
| 공통 UI | `core/ui` | 버튼·텍스트필드·앱바 등 재사용 컴포저블. 로딩 인디케이터·빈 화면·스켈레톤 중 팀 기본 하나 |
| 로딩 UX | `UiState` + 공통 컴포저블 | 전 화면 오버레이 vs 화면별 인라인 로딩 중 기본 한 가지 |
| 에러 UX | `core`의 얇은 타입 + 매핑 | 사용자에게 보일 문구: 예외 메시지 직출력 지양, 네트워크·권한·기타 구분. 스낵바·다이얼로그·인라인 중 기본 |
| Async | (레이어 규칙) | Repository는 `suspend`/`Flow`, ViewModel은 `viewModelScope`, UI는 상태 수집만 — 상세는 [코딩 컨벤션](./코딩-컨벤션.md) §4·§7·§8 |

비즈니스 규칙·화면별 문구는 feature에 두고, `core`에는 정말 여러 화면이 같이 쓰는 최소만 올린다.

`feature` 한 모듈 안 (예: `login`) — `data` / `ui` / `viewmodel`

기능 패키지 안에서 MVVM을 폴더로 나눌 때 흔한 형태다. (`model`·`view` 대신 `data`·`ui`를 써도 된다.) 폴더 역할은 위 표와 같다.

```text
feature/login/
├── data/                 # Repository·모델(기본), 선택 시 DataSource (feature 단위 공유)
├── ui/                   # Compose 화면·컴포저블 (기본 = View)
└── viewmodel/            # 화면별 ViewModel (기본)
```

화면이 많으면 `ui/login/`, `ui/register/`처럼 화면별 하위 패키지를 둔다.

## 3. Firebase (프로젝트 · 앱 · 제품)

Firebase는 프로젝트 하나·`google-services.json` 하나를 기준으로 두고, 쓰는 제품마다 Gradle에 모듈을 추가한다(예: `firebase-firestore`, `firebase-storage`).

콘솔에서 Android 앱 등록·`google-services.json` 내려받기·플러그인 적용·`applicationId` 일치는 [개발 셋업 가이드](./개발-셋업-가이드.md) §4에서 맞춘다.

### 3.1 Firestore

- 데이터·컬렉션 설계는 [Firestore 설계 가이드](./Firestore-설계가이드.md)를 따른다.
- 보안 규칙: 개발 초기와 제출·배포 전에 읽기·쓰기 범위를 다시 점검한다. 테스트용 완전 개방 규칙은 기간을 정해 두고 좁힌다.

### 3.2 Cloud Storage (Firebase Storage)

- 업로드/다운로드는 앱에서 Firebase Storage SDK로 처리하고, 버킷·Storage Rules는 콘솔에서 설정한다.
- 인증이 필요한 경우
  - 보통 `Firebase Auth` 상태를 기준으로 Storage Rules에서 접근을 제한한다(앱 쪽에서 토큰을 직접 넣지 않아도 SDK가 인증 상태를 반영하는 편).
  - 앱은 업로드 요청 시 파일(URI/바이트)와 메타데이터(가능하면 MIME type)를 함께 전달한다.
- URL 처리
  - 파일을 공개 URL로 관리할지, 필요한 순간에만 다운로드 URL을 받아서 사용할지 팀 규칙에 맞춘다.
  - 가능하면 “항상 공개 URL을 저장”하기보다 필요한 때에만 `getDownloadUrl()`로 가져오고, 권한이 필요한 데이터는 규칙으로 통제한다.
- 로컬 개발/테스트
  - 이미지/영상처럼 용량이 큰 파일은 최대 크기·타임아웃을 체크하고, 업로드 실패 시 사용자에게 재시도 UI를 제공한다.

### 3.3 Cloud Functions

- 서버에서 `onCall`(HTTPS Callable)과 `onRequest`(HTTP URL) 등으로 정의할 수 있고, 둘 다 Cloud Functions로 배포된다. 보통은 함수마다 한쪽만 쓰고, 필요하면 같은 로직을 두 방식으로 노출하기도 한다.
- Callable: 앱에서는 `firebase-functions` SDK로 부르는 경우가 많다(같은 프로젝트·`google-services.json`).
- HTTP URL: Retrofit/OkHttp 등 일반 HTTPS 클라이언트로 부른다. Base URL·인증은 §4 “Cloud Functions(HTTP) 호출”을 본다.

검증

- 앱에서 로그인(해당 시)·Firestore 읽기/쓰기, (해당 시) Storage 업로드/다운로드 한 번씩 성공하는지 확인한다.

## 4. 기타 백엔드·외부 서비스

REST API, Supabase, Cloud Functions의 HTTP URL처럼 Firebase 제품 SDK와는 별도로 Base URL·키를 빌드 설정으로 넘기는 연동이다(Supabase DB는 [Supabase 설계 가이드](./Supabase-설계가이드.md)):

| 항목 | 정리 방법 |
|------|-----------|
| Base URL | `debug` / `release`별로 다르면 `buildTypes` 또는 `BuildConfig`로 분리 |
| API 키 | 저장소에 넣지 말고 `local.properties` 등 로컬 전용 파일 → `BuildConfig` 주입 패턴 검토 |
| 환경 | 스테이징·프로덕션 주소 혼동 방지를 위해 빌드 변수명을 명확히 할 것 |

Cloud Functions(HTTP) 호출

- §3.3에서 말한 HTTP URL(`onRequest` 등)을 앱에서 부를 때의 절차다. Callable SDK 말고 일반 HTTP 클라이언트를 쓴다.
- 엔드포인트(Base URL + path)는 앱 설정 값으로 빼 둔다. 예: `https://<region>-<project>.cloudfunctions.net/<functionName>`
- 인증이 필요한 경우(대부분 권장)
  - 앱에서 사용자 ID 토큰을 받아 `Authorization: Bearer ...` 형태로 붙인다(팀 규칙에 따름).
  - 검증·권한은 Functions 쪽에서 Admin SDK로 처리한다.
- 로컬 개발에서는 팀에서 정한 방식으로 동작(실배포 URL 그대로 호출 / Firebase Emulator Suite 사용 등)을 선택한다.

`local.properties`·`BuildConfig` 연동 예시는 [개발 셋업 가이드](./개발-셋업-가이드.md) §5 부록을 본다.

## 5. 트러블슈팅

Gradle·Firebase·에뮬 쪽에서 자주 나오는 것만 골라 두었다. 막히면 Logcat·Gradle 창에서 에러 한 줄 복사해 검색해 보면 된다.

| 증상 | 자주 있는 원인 | 조치 |
|------|----------------|------|
| Gradle Sync·빌드 실패(`OutOfMemoryError` 포함) | JDK 불일치, 프록시, 캐시, AGP/Kotlin/KSP 버전, JVM 힙 | Gradle JDK를 프로젝트 권장값으로. `./gradlew --stop` 후 Sync. `libs.versions.toml`·`build.gradle.kts`가 main과 동일한지, Clean/Rebuild. 캐시 의심 시 Invalidate Caches. OOM이면 `gradle.properties`의 `org.gradle.jvmargs`로 힙 확대 |
| `SDK location not found` | `local.properties`·`sdk.dir` | 프로젝트 루트에 `local.properties`와 올바른 `sdk.dir`이 있는지 확인 |
| `Duplicate class` / 의존성 Conflict | 같은 라이브러리 중복 버전 | Firebase는 BOM으로 맞춤. `./gradlew app:dependencies`로 트리 확인 후 중복 `implementation` 정리 |
| `google-services.json` 오류·머지 후 Firebase만 깨짐 | 경로·패키지 불일치, 충돌 마커 잔존 | `app/`(또는 Firebase 모듈 루트) 배치, 콘솔 패키지명과 `applicationId` 일치. `<<<<<<<` 남았으면 제거하거나 콘솔에서 json 새로 받기 |
| Firestore `PERMISSION_DENIED`·Storage 거부·Auth 로그인 실패 | Rules, 비로그인, 제공자 미활성, 미인증 메일 | Firestore Rules·Storage Rules·Authentication → Sign-in method 확인. 로그인 uid·문서 경로·`ref` 경로. 이메일 링크 인증이면 스팸함까지 |
| 설치 거부(`INSTALL_FAILED_…`) | 다른 키로 서명된 동일 앱 잔존 | 기기에서 앱 삭제 후 재설치. `applicationId`가 팀원과 같으면 한 기기에 하나만 |
| 실행은 되는데 하얀 화면·NPE | Nav/ViewModel 주입 실패 | Logcat 스택 확인. `NavHost` 경로, `hiltViewModel()` 스코프, Nav 인자 누락 |
| 에뮬에서만 지도·FCM·네트워크 이상 | Play 없는 이미지, DNS 등 | Google Play 포함 시스템 이미지로 새 에뮬. Cold boot·다른 이미지·실기 재현 여부로 원인 좁히기 |

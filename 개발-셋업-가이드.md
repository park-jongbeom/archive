# 개발 셋업 가이드 for AOS 6기

PC·Android Studio·저장소·빌드와 Firebase **앱 등록·json·플러그인**까지 다룬다. 아키텍처·Firestore/Storage/Functions 상세·트러블슈팅·팀 운영은 [개발 가이드](./개발-가이드.md)를 이어서 본다.

---

## 1. 개발 환경 (로컬)

| 항목 | 권장 / 참고 |
|------|-------------|
| OS | Windows / macOS / Linux (팀원과 동일한 JDK 권장) |
| JDK | Android Studio가 번들하는 JDK 사용, 또는 프로젝트가 요구하는 버전과 일치 |
| Android Studio | 최신 Stable 권장. 설치 후 SDK Manager에서 필요한 플랫폼·도구 설치 |
| Git | `git --version`으로 설치 확인 |

에뮬레이터 vs 실기

| 구분 | 비고 |
|------|------|
| 에뮬레이터 | Google Play가 포함된 시스템 이미지 사용 시 Play 서비스·일부 Firebase 동작 확인에 유리 |
| 실기 | USB 디버깅 활성화, 동일 Wi‑Fi에서 로그·네트워크 확인에 유리 |

프로젝트별 `minSdk` / `compileSdk` / `targetSdk`는 각 앱 모듈의 `build.gradle.kts`에서 확인한다.

---

## 2. 저장소·협업

브랜치·PR·이슈·마일스톤 흐름은 [GitHub Flow 가이드](./github-flow-브랜치-전략.md)를 따른다.  
PR·커밋 템플릿은 같은 문서의 “PR 템플릿·커밋 메시지 (선택)” 절과 [templates/](./templates/) 샘플을 본다.

| 항목 | 할 일 |
|------|--------|
| 클론 | 팀에서 안내한 URL로 저장소 복제 |
| 브랜치 | `main`에서 기능 브랜치 분기 → PR로 병합. 네이밍·워크플로 상세는 위 링크 |
| PR | 리뷰 필수 여부, 담당자 지정 규칙 준수 |
| 이슈·마일스톤 | 작업 단위는 Issues로 등록하고, 중간·제출 목표는 Milestones으로 묶는다. 브랜치·PR과 번호로 연결([GitHub Flow 가이드](./github-flow-브랜치-전략.md) 참고) |

커밋에 넣지 말 것

| 파일·유형 | 이유 |
|------------|------|
| `local.properties` | 로컬 SDK 경로 등 환경 의존 |
| `google-services.json` (팀 정책에 따라) | Firebase 설정. 공개 저장소면 반드시 제외하고 비공개 배포 채널 사용 |
| 키스토어(`.jks` / `.keystore`) | 앱 서명 키 유출 방지 |
| API 키·비밀 문자열이 하드코딩된 파일 | `BuildConfig` + 로컬만 두는 값으로 분리 |

`.gitignore`에 위 항목이 포함되어 있는지 PR 전에 확인한다. 부록에 `.gitignore` 줄 예시가 있다.

## 3. Android 프로젝트 열기 및 빌드

1. 열기: `settings.gradle.kts`(또는 `settings.gradle`)가 있는 프로젝트 루트를 Android Studio에서 연다.
2. Gradle Sync: 상단 알림 또는 File → Sync Project with Gradle Files. 실패 시 JDK·프록시·인터넷 확인.
3. 빌드 확인  
   - Studio: Build → Make Project  
   - 터미널(프로젝트 루트): `./gradlew assembleDebug` (Windows: `gradlew.bat assembleDebug`)

셋업 완료 최소 기준: Debug 빌드 성공, 에뮬 또는 실기에서 실행 확인.

## 4. Firebase 최소 연동 (해당 시)

팀이 Firebase를 쓰면 아래를 맞추면 앱에 SDK를 붙이고 빌드할 수 있다. Firestore·Storage·Functions·보안 규칙 등 **제품별 사용**은 [개발 가이드](./개발-가이드.md) §3을 본다.

| 단계 | 내용 |
|------|------|
| 콘솔 | Firebase 프로젝트 생성 또는 팀 프로젝트에 멤버로 초대받기 |
| Android 앱 등록 | 패키지명(`applicationId`)이 `build.gradle.kts`의 `defaultConfig`와 문자 단위로 동일해야 함 |
| `google-services.json` | Firebase 콘솔에서 내려받아 앱 모듈(보통 `app/`)에 배치. 경로는 [공식 문서](https://firebase.google.com/docs/android/setup) 기준 |
| 플러그인 | 루트·앱 `build.gradle.kts`에 Google Services 플러그인 적용 여부 확인 |

## 5. 부록: 경로·설정 샘플

아래는 형식 참고용이다. 경로·키 이름은 프로젝트와 팀 규칙에 맞게 바꾼다. 실제 비밀 값은 문서나 저장소에 넣지 않는다.

### 5.1 프로젝트 루트 디렉터리 예시

```text
MyApplication/
├── settings.gradle.kts
├── build.gradle.kts
├── gradle/
├── gradlew
├── local.properties          ← 보통 .gitignore (커밋 제외)
├── app/
│   ├── build.gradle.kts
│   ├── google-services.json  ← 팀 정책에 따라 커밋 여부 결정
│   └── src/main/...
└── ...
```

### 5.2 `local.properties` 예시

Android Studio가 만든 `sdk.dir` 외에, 팀에서 허용한 경우에만 API 관련 키를 둔다.

만약 팀에서 `.env`로 환경변수를 관리하기로 했다면, Android/Gradle에서 기본적으로 자동 로딩되지는 않으므로 Gradle 설정에서 `.env` 값을 읽어 `BuildConfig` 또는 리소스로 주입하는 방식으로 처리해야 한다.

```properties
# SDK 위치 (macOS/Linux 예 — 본인 PC 경로로 다름)
sdk.dir=/Users/사용자이름/Library/Android/sdk

# Windows 예 (백슬래시 이스케이프)
# sdk.dir=C\:\\Users\\사용자이름\\AppData\\Local\\Android\\Sdk

# 팀 규칙으로 여기에만 둘 때 (이름은 예시)
# API_BASE_URL=https://api.example.com
# MY_SECRET_KEY=실제값은여기에만
```

### 5.3 `.gitignore`에 자주 넣는 항목 예시

```gitignore
local.properties
*.jks
*.keystore
```

공개 저장소라면 `google-services.json`도 제외하는 팀이 많다. 팀에 맞게 한 줄 추가한다.

```gitignore
# 팀 정책에 따라
# app/google-services.json
```

### 5.4 `applicationId` 확인 (`app/build.gradle.kts`, Kotlin DSL)

Firebase 콘솔에 등록한 패키지명과 문자 단위로 동일해야 한다.

```kotlin
android {
    defaultConfig {
        applicationId = "com.example.myapp"
        minSdk = 24
        // targetSdk, compileSdk는 팀·프로젝트에서 정한 값 사용
    }
}
```

### 5.5 `local.properties` → `BuildConfig` 주입 예시 (개념)

`app/build.gradle.kts`에서 루트의 `local.properties`를 읽어 빌드 시점에 필드를 넣는 패턴 예시다.

`local.properties` (로컬에만 존재):

```properties
API_BASE_URL=https://dev-api.example.com
```

`app/build.gradle.kts` (발췌):

```kotlin
val localProperties = java.util.Properties().apply {
    rootProject.file("local.properties").takeIf { it.exists() }
        ?.inputStream()?.use { load(it) }
}

android {
    defaultConfig {
        buildConfigField(
            "String",
            "API_BASE_URL",
            "\"${localProperties["API_BASE_URL"] ?: ""}\""
        )
    }
    buildFeatures {
        buildConfig = true
    }
}
```

코드에서는 `BuildConfig.API_BASE_URL`로 접근한다. 키 이름·모듈 구조는 팀과 맞춘다.

| 순서 | 할 일 |
|------|--------|
| 1 | [전제와 용어](#2-전제와-용어) 회의 |
| 2 | [F1~F5](#3-f1f5-역할)에 실제 기능 이름 대입 |
| 3 | [현실안 두 가지](#5-현실안-두-가지) 선택 후 **책임자 표**에 이름 기입 |

## 2. 전제와 용어

| 항목 | 내용 |
|------|------|
| 인원 | **5명 기준**(4명·6명 등이면 F3/F4 범위만 조정해 같은 원칙 적용) |
| 우선순위 | **F1 인증** 먼저 권장 (로그인 상태·진입 흐름) |
| 데이터 | **목 데이터**로 UI·동작 먼저, 이후 실연동 |
| F5 | 급하지 않으면 후순위. 시연용 **로그아웃 등 최소**만 정하기 |
| 완료 단위 | 이슈마다 완료 조건 + PR로 `main` 병합 |

| 용어 | 의미 |
|------|------|
| 목(mock) | DB 대신 쓰는 가짜 데이터 |
| 계약 PR | 로그인 상태·라우팅 등 **다른 사람이 기대해도 되는 규칙**을 짧게 고정한 PR |
| Repository | feature `data`에 **기본**으로 두는 데이터 창구(ViewModel이 호출). 목/실을 나눌 때 DataSource는 **선택**. ([개발 가이드](./개발-가이드.md) **§1**, 아래 **§4.3** 표) |

## 3. F1~F5 역할

역할 이름이며, 기능 정의서의 **실제 메뉴명**으로 바꿔 씁니다.

|  | 역할 | 범위 예시 |
|--|------|-----------|
| **F1** | 인증·세션 | 가입·로그인·로그아웃·전역 로그인 상태·진입 흐름 |
| **F2** | 앱 골격·네비 | 패키지·탭/스택·공통 테마·**빈 화면 자리** |
| **F3** | 도메인 A | 팀 고유 기능 1 (예: 게시판) |
| **F4** | 도메인 B | 팀 고유 기능 2 |
| **F5** | 마이페이지·설정 | 프로필·설정·로그아웃 진입 등 |

**필드 이름**은 기능 정의서/API와 맞춰 두면 목→실연동 시 유리합니다.

## 4. 권장 순서, 이슈, 아키텍처

### 4.1 큰 순서

1. F1에서 로그인 상태·규칙 짧게 고정 → **계약 PR**  
2. F2에서 네비·탭·빈 destination  
3. F3·F4 목으로 화면·동작 → 이후 Firestore 등  
4. F5는 여유 있으면 (없으면 최소만)

### 4.2 세부 이슈 (공통)

“게시판 한 방”이 아니라 **목록 / 상세 / 등록 / 수정 / 삭제** 등으로 이슈 분할. 한 이슈 ≈ 한 PR 크기.

| 축 | 예시 |
|----|------|
| 화면 | 목록·상세·작성·수정 화면 |
| CRUD | 생성·조회·수정·삭제·검색 |
| 병렬 시 | **서로 다른 화면·ViewModel** 이슈로 나눔. **같은 Repository(또는 선택적 DataSource) 파일**을 동시에 크게 수정하지 않기 |

F1도 가입 / 로그인 / 세션 등으로 쪼갤 수 있음.

### 4.3 코드 구조 ([개발 가이드](./개발-가이드.md) MVVM + Feature-based)

|  | 단위 | 원칙 |
|--|------|------|
| **ViewModel** | 화면(또는 화면 단위 플로우) | 기능 화면이면 **화면 : ViewModel ≈ 1:1** 권장 |
| **Repository** | feature(F) `data` **기본** | ViewModel이 부르는 데이터 창구. **화면마다 새로 만들지 않고** 같은 feature에서 공유. Firestore 호출은 여기서 직접 해도 됨 |
| **모델(DTO 등)** | feature(F) `data` **기본** | 이 기능 전용 타입은 **Repository와 같은 `data` 패키지**에 둔다(`model` 하위 폴더는 팀 선택). |
| **DataSource** | **선택** | 목/실Firestore·출처별로 **클래스를 나눌 때만**. 없으면 Repository만으로 진행 ([개발 가이드](./개발-가이드.md) §1 참고) |
| **이슈 묶음** | UI + ViewModel | Compose는 상태·로딩·에러를 UI와 같이. 목은 **ViewModel** 또는 **Repository(또는 선택적 DataSource)** 에 두고 이후 실연동만 교체. **인터페이스 필수 아님** |
| **복수 인력이 같은 F** | 리드 = F 책임자 | **`data`(Repository·모델)** 뼈대 PR을 리드가 먼저(DataSource를 쓰는 팀은 그 PR에 포함) → 나머지 화면 이슈 병렬 |
| **Repository·모델 변경** | F 책임자와 합의 | 해당 feature 화면 전부에 영향(DataSource를 쓰면 그 변경도 동일) |

## 5. 현실안 두 가지

|  | ① 안정형 | ② 속도·유연형 |
|--|----------|----------------|
| 요약 | F당 책임자 1명, 세부는 이슈로 순차 | 초기에만 일부 F에 **보조** 붙임(이슈 단위). 이후 수렴·**재배치** 가능 |
| 장점 | 역할 설명 쉬움, 충돌 적음 | 초반 병렬·인력 이동(F4/F5 보강 등) |
| 주의 | F 크면 병목 → 이슈 세분화 또는 ② 혼합 | 이슈 분할·**짧은 머지** 없으면 역효과 |
| 추천 | 처음·단순 운영 | 초반 압박·F1·F2 후 F4/F5로 돌리고 싶을 때 |

**공통:** 아래 표는 ①·② **동일**하게 사용 (책임자 = 리드로 두어도 됨).

**기능별 — 책임자 이름**

| 기능 | 맡는 일 | 이름 |
|------|---------|------|
| F1 | 인증·세션 | |
| F2 | 골격·네비 | |
| F3 | 도메인 A | |
| F4 | 도메인 B | |
| F5 | 마이페이지·설정 | |

**멤버별 — 대표 F**

|  | 이름 | F |
|--|------|---|
| 멤버 1 | | F1 |
| 멤버 2 | | F2 |
| 멤버 3 | | F3 |
| 멤버 4 | | F4 |
| 멤버 5 | | F5 |

### 안 ② — 배치 예시 (5명, 숫자는 팀이 정함)

#### 초기 인원 (한눈에)

| F | 투입 | 한 줄 메모 |
|---|------|------------|
| **F1** | 1명 | 인증·세션 |
| **F2** | 1명 | 네비·빈 화면 자리 |
| **F3** | **2명** 권장 | **이슈로만** 나눔. **책임자 1 + 보조 1**. 리드가 **Repository·모델 뼈대 PR** 먼저(선택적 DataSource 포함) |
| **F4** | 1명 | 도메인 B |
| **F5** | 0명(일반) | 시연용 로그아웃 등 → **F1/F2 이슈**에 포함 |

#### 변형 (같은 ② 안)

- **F3에 3명** · F4·F5는 **초기 0명** → `F1+F2+F3 = 5명`으로도 가능.  
- 위와 동일하게 **리드 선행 Repository·모델** · **이슈 분할** 적용.

#### 이후 재배치

| 예시 | 내용 |
|------|------|
| F1·F2 | 주 업무 끝나면 → **F4 보강** 등으로 이동 |
| F3 팀 | 일부·전체 → **F5(마이페이지)** 착수 등으로 이동 |

- **언제 옮길지**는 마일스톤·팀 메모에 적기.  
- “완전 종료”보다 **주 담당만 옮기고**, 인증·네비 **긴급**은 합의 하에 대응.

#### 시작 팁

F1·F2가 **계약 PR**을 먼저 올리면, F3·F4 **목 작업**이 수월함.

## 7. 팀 규칙

| 규칙 | 요지 |
|------|------|
| 이슈·브랜치 연결 | PR에 이슈 번호·`Closes #n` |
| 규칙 변경 공유 | 인증·네비 변경 시 F3·F4 목 스키마와 충돌 주의 |
| 자주 `main` 합침 | 장기 브랜치보다 작은 PR |
| 발표 시나리오 1줄 | 그 외 기능은 이슈로 미루기 합의 |

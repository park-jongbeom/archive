## 0. 상담의 도입

정원화가 보조강사 자리에 직접 찾아와 시작. 분위기는 처음부터 가벼웠고, 정원화 본인이 "오늘 강사님을 독차지하러 왔다"고 표현할 만큼 적극적이었음.

- **정원화 첫 발화**: *"저는 기획자 포지션이 된 것 같아요"*
- **상담 동기**: *"지금 이게 하고 있는 게 맞긴 한가 그런 느낌이 들거든요. 일단 저희 팀은 안전감을 느끼고 있긴 해요"* — 안전감은 있으나 방향이 맞는지 검증받고 싶음
- **본인 자기 인식**: *"저는 사실 나이가 조금 있기도 하고 해서 그런지 다른 팀원들 보다는 확실히 좀 그런 거에 약간 강박적이긴 하거든요. 어떤 조직에서 원하는 템플릿이 있고 어떤 조직에서 원하는 작업의 형식이 있으면 거기에 맞추는 게 굉장히 중요하다고 생각을 많이 하거든요"* — 권위(강사 가이드)와의 align 욕구가 강함

## 1. 주제 A — 백로그 / 문서 작업의 방향성 (대화의 약 1/3)

### 1-1. 정원화가 가져온 고민

- 현재 본인이 **팀원들보다 1~2 스텝 선행해서** 문서 작업 진행 중 (기획·설계 영역)
- 팀원 작업이 끝나기 전에 다음 작업 준비 / 공통 작업 선행 필요 → 본인이 선행 → 재민(팀장) 작업 → 나머지 팀원 작업, 이런 순서를 의도
- 하지만 강사 가이드가 일관되지 않게 느껴짐:
  - *"갑석강사님이 이렇게 해야 된다 저렇게 해야 된다라는 말씀을 중간중간에 해주실 때 사실 그걸 그래서 어떻게 해야 되는 건지 잘 이해가 안 되는 부분들이 있거든요"*
  - *"약간 일관성이 안 느껴지는 부분들도 있기도 하고요"*
- 구체적 혼란 지점: GitHub 이슈 템플릿이 "백로그 + 기능명세 역할" 이라고 적혀 있음 → 그러면 본인이 "명세서에 가깝게 상세하게" 작성해 가는 게 맞는 줄 알았는데, 강사가 5/15 PM에 "이슈 단위를 더 가볍게 쪼개라"고 함 → **어디에 align해야 하는지 모르겠음**
- 본인이 작성한 문서들 = 백로그에 올리는 명세서? 아니면 별도 위치?

### 1-2. 보조강사 조언

**(1) Just-In-Time 문서화 원칙**
> *"문서도 보통 그 바로 다음 자추나 이제 그거에 필요한 문서만 먼저 빠르게 뽑아내서 공유를 하는 식으로 하거든요"*

> *"원칙적으로 따지면 원하님께서 기획하시는 그런 문서들은 사실 개발하기 전에 원래 나와 있어야 되는 게 정상이죠"* — 즉 정원화의 선행 작업 방향 자체는 정당, 다만 분량·범위 조절 필요

**(2) 백로그의 실제 구조 — Epic / Story / Task**
- 정원화 작업물(OS001, DASH001 등)은 **"그루핑 단위"** = Jira의 Story 수준
- 실제 백로그는 작업 단위 (Task)
- 로그인 기능 예시:
  - 로그인 UI 작업
  - 로그인 데이터 모델링 작업
  - 로그인 API 연동 작업
  - (단위 테스트까지 하면) 단위 테스트 작업
  - → **각 팀원이 본인 작업 단위로 직접 백로그에 등록**
- 클린 아키텍처 레이어(View / VM / Domain)와 작업 단위 분리가 정합

**(3) 정원화 작업물 가치 인정 ⭐**
> *"이 정도까지 만들어서 주시면 사실 팀원들 입장에서는 너무 좋죠"*
> *"이게 팀장급이 해야 되는 업무거든요"*
> *"개발한 일정에서는 아주 좋거든요"*

→ 정원화가 작성한 그루핑 단위 문서(상세 플로우)는 **버리지 말고 유지**. 다만 그 하위 작업 단위는 팀원들이 본인이 직접 짜서 백로그에 등록하는 식으로 분리.

**(4) 합의된 운영 방안**
- 정원화 → 그루핑 단위 / 흐름 / 완료 기준 문서화 (현행 유지)
- 각 팀원 → 본인 작업 단위로 백로그 직접 등록 + 참조할 문서 위치 백로그에 표기
- *"어느 수준까지 이렇게 정리해서 주실 건가는 좀 얘기를 해보셔야 될 것 같아요"* — 팀 합의 필요 (정원화의 작업 범위 정의를 팀원과 함께)

### 1-3. 정원화의 수용

> *"이해가 많이 된 것 같아요. 그러니까 이제 저는 달팡질팡 그래서 어떻게 하라는 거지 막 했던 거거든요. 그래서 지금 이 그루핑 단위로 제가 문서 작업을 한 거예요. (...) 근데 이거를 그루핑 단위의 문서니까 필요한 거고, 사실 이거 외에 더 작은 기능 단위로 이슈를 뭘 이렇게 해야 된다라는 걸 이제 말씀해 주셨으니까 이제 방향성을 알겠는 거죠"*

→ 1차 매듭. 그루핑 문서는 유지, 작업 단위 백로그는 팀원에게 위임이라는 결론.

## 2. 주제 B — 시각화·다이어그램의 필요성 + AI 활용 실습 (대화의 약 1/3)

### 2-1. 정원화가 가져온 고민

- 본인 작성 문서가 상세함은 충분하나 **"파악하는 시간이 오래 걸리는"** 문제
- 워터폴 시절 ERD·화면 명세서 같은 **한눈에 보이는 시각적 자료**의 부재 아쉬움
- 하지만 시각화까지 다 작업하면 *"애자일 방식이 맞나"* 라는 갈등
- 본인의 도식화·AI 활용 경험은 제한적 — *"해본 적이 없긴 하거든요. 어떤 퀄리티로 이해하고 잘 뽑아보는지 결과물을 받아본 적이 없어서요"*

### 2-2. 보조강사 조언 + 실습 시연 ⭐

보조강사가 **직접 VSCode + Codex CLI를 띄워서 정원화 문서를 입력해 다이어그램 생성 시연**.

**(1) 도구 추천**
- **구글 드라이브 다이어그램** — 무료, 처음 보는 사람이 많음. 직접 수정 가능
- **SVG 형태로 출력** 권장 → 이미지로 만들면 수정 불가. SVG는 Figma/Stitch에서 복붙하여 재가공 가능
- **Figma** — 디자인 작업용, MCP 연동 시 코드↔디자인 가능
- **Stitch (Google)** — Figma 대안, 부분 수정 편함. 재훈이 웹에서 사용 중이라는 언급

**(2) AI 모델 운용 노하우 — 토큰 절약 ⭐**
> *"GPT 5.5는 전반적인 큰 분석이나 검토에 한번씩, 그 다음에 5.4 미니로 세부 작업을 쭉쭉 하고, 왔다 갔다 하면서"*

- 5.5 (큰 분석/검토) ↔ 5.4 mini (세부) 의도적 왕복
- Claude Code Max ($200/월) 도 사용 중이나 limit 때문에 풀로는 못 씀
- *"본인이 또 수정하면 되거든요. 그러면 굳이 AI한테 명령 안 하고 토큰을 줄일 수가 있습니다"*

**(3) 도식화가 필요한 영역 식별**
- 단순 플로우(예: 화면 전환)는 도식화 불필요
- **로컬 데이터 ↔ 파이어스토어 sync 흐름** = 반드시 도식화 (논리적으로 끊겼을 때 원복 로직 등)
- *"로직을 순서로 정리해서 주면 이해가 좀 쉽겠죠"* — 팀원들이 한 번에 이해 어려운 부분만 선별 도식화

### 2-3. 정원화의 수용

> *"그러니까 너무 좋은 시간입니다. 저는 독차지 하려고 왔습니다 오늘 강사님을"*
> *"피그마라도 할 수 있겠네요. 연결해가지고 그죠"*
> *"스티치 한번 써보시면 되게 편합니다 (라고 들었으니 써봐야겠다)"*

→ 시각화 도구·AI 활용 모두 본인 영역으로 적극 수용. 부담 X.

## 3. 주제 C — 하네스 엔지니어링 / AI 다루는 법 (대화의 약 1/6)

### 3-1. 정원화가 가져온 관심

- *"AI 다루는 여러 가지 방법이나 이런 것들 자체에도 익숙해 주는 게 좋겠다라고 생각은 하고 있죠"*
- 본인은 부트캠프 초중반은 웹에서 작업, 플러터 미니 프로젝트 때부터 agent 도입 시작 — 새로운 것 조금씩 도입 중
- 본인 AI 사용 원칙: *"AI가 뭔가 만들어주는 코드나 이런 것들 절대 이해 안 하고 넘어가면 안 돼요. 일단 기본적으로 믿지 않거든요. 그래서 다 읽어봅니다"*

### 3-2. 보조강사 조언

**(1) AI 트렌드 변천**
> *"프롬프트 엔지니어링 → 컨텍스트 엔지니어링 → 하네스 엔지니어링 이런 식으로 트렌드가 좀 넘어왔거든요"*

- Prompt: AI에게 명확하게 지시
- Context: 내부 데이터·언어를 저장해 자동 적용
- **Harness**: "안 해야 되는 것 / 지켜야 되는 것"을 명확히 정의 = AI 행동을 규칙으로 제어

**(2) 보조강사 본인의 5단계 워크플로우 ⭐**
1. **계획 단계** — AI에게 계획부터 시킴 + 모델·토큰 추천 받음
2. **구현 단계** — 계획대로 코딩 진행 지시
3. **자가 검증** — 코딩 완료 후 AI에게 본인 코드 검증 시킴
4. **리팩터링** — 문제 있으면 수정, 사람이 같이 체크
5. **문서화** — 작업 내용 정리 → 다음 작업 인계용

**(3) 정원화 운용 평가 ⭐**
> *"제가 봤을 때 원화님은 아주 잘 사용하고 계신 겁니다. 그리고 본인 스스로 하네스 기법을 잘 적용하고 계신 거고요"*

- 본인 강의 자료 / 정본 강사 강의 자료 채팅창에 공유 (스타터팩 / 5-레이어 규칙)
- *"본인만의 스타일로 작성을 해야 하는 거거든요"* — 정원화의 목표(문서 위주)에 맞춰 규칙 정의 권장

### 3-3. 정원화의 수용

> *"용어에서도 살짝 느껴지는데 약간 제한을 두고 방향성을 제시하는 느낌인 것 같더라고요"*
> *"AI 다루는 여러 가지 방법이나 이런 것들 자체에도 익숙해 주는 게 좋겠다"*

→ 부트캠프 종료 후 본인 창업 활동에 직결되는 영역으로 인식 + 수용.

## 4. 주제 D — 창업 / 부트캠프 종료 후 계획 (대화의 약 1/6)

### 4-1. 정원화가 가져온 계획

- 부트캠프 끝나면 **몇 개월 여행 + 작은 서비스 개발**
- 본 BM(언어 학습) = 본인 창업 아이템:
  - MVP는 "한국인 영어 학습"
  - 진짜 타겟은 **"한국어 배우는 외국인"** (글로벌 시장, 경쟁 적음)
  - 궁극은 다국어 학습 모델
- 부트캠프에서 안 배운 영역 학습 필요 인지:
  - 결제(페이) 연동
  - 보안
  - 어드민 (서비스 관리)
  - 배포 인프라
- 본인 전략: **"작은 서비스를 먼저 많이 만들어보자"**
- 웹 중심 선호 (*"스마트폰이라는 기기는 트렌드가 끝날 수 있다고 생각"*)

### 4-2. 보조강사 조언

**(1) 비용 / 자금 조달** ⭐⭐
> *"절대 본인 돈으로 하면 안 되더라고요. 그래서 무조건 정부 지원이나 이런 것들을 통해서 받을 수 있는 것을 창출을 해야 되는데"*

- "모두의 창업" 등 지원사업 정보 추적 권장
- **아이디어 변경 X, 출제처 입맛에 맞춰 구색 조정** — 교육부(언어교육) / 관광청(한국 방문 전 학습) / AI 공모전 등 동일 아이템 multi-positioning
- 본인 경력 언급: 예술가 풀 / 도시공헌화 사업 10억 단위 PM 경험

**(2) 사업 아이템 평가**
> *"잘 팔릴만한 아이템이라고 생각합니다"*
> *"사업적인 아이템으로 봤을 때는 좋았거든요"*

- API 비용 구조 사전 설계 강조 — 무료 사용자/유료 사용자 분리
- 손익분기점 = 월 발생 비용 최소화 = 자체 백엔드 구축 필요

**(3) 기술 학습 권장** ⭐
> *"백단을 하셔야 될 거예요. 왜냐하면 지금 만들려는 이 아이템도 결국은 돈이 안 들어가게끔 하시려면 백단을 배우셔가지고 자체적으로 구축을 하셔야 돼요 서버 쪽을"*

- Firebase 의존 → 자체 서버 구축으로 전환 시 학습 필요
- 웹 + 백엔드 같이 (React 등 프론트 배우면 서버 학습 자연 동반)
- 국비 지원 한도 내 추가 부트캠프 가능성 조사 권장

**(4) 작은 서비스 전략 격려**
> *"그렇게라도 만들어주는 서비스면 성공적인 서비스죠"* (수익 100만원이라도)
> *"올해 안에 뭔가 한번 수익을 내는 서비스를 작은 거라도 (...) 충분히 가능하실 것 같습니다"*

### 4-3. 정원화의 수용

> *"이해가 됩니다. 그렇군요. 올해 안에 뭔가 한번 수익을 내는 서비스를 작은 거라도"*
> *"그런 공부도 해봐야겠네요"* (백엔드)

## 5. 주제 E — 재훈(팀원) 케어 + 팀 운영 상담 (대화의 약 1/6)

### 5-1. 정원화가 가져온 팀 상황 인식 ⭐

> *"강사님들이 팀을 짜실 때 엄청 고심을 많이 하셨을 텐데 재훈님이 왜 우리 팀에 왔는지에 대해서 생각해봤다. 내가 생각하기에 재훈님이 우리 팀에 배정된 이유는 당신 거 나 때문이다"*
> *"우리의 목표는 재훈님이 프로젝트 내에서 이탈되지 않도록 하고 (...) 절대 배제시키지 말고"*

→ 정원화 본인이 재훈 케어를 자기 미션으로 명시적으로 받아들이고 있음.

**재훈 지각 사건(언급된 일):**
- 지각 → 마이크 끄고 아무 말 없이 1시 등장 → 정원화가 "세게" 1:1 + 공개적으로 지적
- 발화 인용: *"먼저 죄송하다라든지 늦어서 죄송하다 무슨 일이 있었다. 그리고 오전에 무슨 내용이 있었는지 너무 죄송하지만 한 번만 브리핑해 볼게요 — 이런 얘기를 먼저 주도적으로 해야지"*
- 정원화 본인 정리: *"감정적으로 속에다 담아 두고 그러고 싶지 않고, 다른 팀원들한테 물어보지는 않았지만 저랑 비슷한 감정을 느낄 것 같고, 털고 가야 앞으로 원활하지 않겠냐"*
- 그 후 *"농담도 해가면서 우서가면서 작업하려고"* 분위기 회복 의도

**재민(팀장) 케어:**
- 재민의 야간성 패턴(새벽 1시 작업) → 정원화가 "팀장 새벽 폭주 절대 금지"를 본인 미션화
- 부팀장 자처 동기: *"재민님을 바디팀장으로 앉혀놓고 비서실세로서 제가 활동한다"*

### 5-2. 보조강사 조언 + 인정 ⭐⭐⭐

**(1) 정원화 부팀장 역할 공식 인정**
> *"원하님께서 지금 중간에서 아주 자리를 딱 잘 잡고 계십니다. 사실 그게 팀장의 역할이거든요"*
> *"부팀장으로서 충분한 역할을 해주신 것 같고요"*

**(2) 2팀 구성에 대한 강사진 인식 공유**
- 2팀 = 재민 빼고 모두 부트캠프 개발 입문자
- 3팀 = 잘하는 사람 모은 "밸런스 붕괴 팀"
- 2팀에 재훈이 배치된 것은 강사진 의도 있음

**(3) 재훈 진척 인정 — 작은 신호 포착 ⭐**
> *"라우터 쪽에 파일을 건드리셨단 말이에요. 근데 그게 꼭 굳이 필요 없는 부분이라고 생각이 드는데 '아 이쪽 부분을 제가 건드리면 안 될 것 같아서 여쭤봅니다'라고 얘기를 하신 것 자체가 발전하신 거라고 생각하거든요"*

> *"재민이랑 코드 리뷰할 때 재민이 질문하기를 '이 카드에서 동적인 부분과 정적으로 출력되는 부분 아시냐'고 했는데, 재훈이 '아 네 동적인 게 저거 말씀하시는 거 보니까' 아신다는 것 자체가 나아진 거"*

→ **재훈에게 임무 완수보다 자기 학습 과정을 돕는 게 목표** (보조강사 명시)

**(4) 재훈 태도 문제 재발 시 에스컬레이션 경로**
> *"이런 일이 발생하면 (...) 강사진들이 계획이 있는 것 같습니다. (...) 매니저님에게 이야기해서 가이드를 받아서 진행을 해봐야 될 것 같거든요"*
> *"여러분들이 같이 팀 프로젝트를 하기 위해서 모인 거지, 교육을 해 주기 위해서 모인 게 아니잖아요. 그런 분들은 멈쟁이 회사 측에서 제재해 줘야 되는 부분이 맞을 것 같거든요"*

→ 정원화에게 책임 떠넘기지 말 것 + 재발 시 정식 경로 안내.

**(5) 재훈 주말 학습 권장 사항 (정원화가 재훈에게 전달할 메시지)**
- 가시적 작업물 남기기 — 코드가 틀려도 됨, 리뷰·가이드 가능한 수준이면 됨
- 사전 지식 / 팀 프로젝트 이해 메모해서 기록 남기기
- 월요일 됐을 때 거리감 벌어지지 않도록 주말 소통 신경

### 5-3. 성우(중도이탈) 언급

- 정원화: *"또 이탈한 사람이 없도록 하겠습니다"*
- 보조강사: *"그거는 원하님께서 어떻게 할 수 있는 일이 아니니까. 하는 데까지 해보겠습니다"*

📜 5/18 1차 검증 Critical 2건 — 정정 사유 상세

1. d1 온보딩 — pending sync 회수 경로 부재 철회

5/18 1차 주장: d1에 Local 1차 저장 → Firestore 배치 저장 단방향만 있고, Firestore 실패 시 pending sync 회수 경로가 표시되지 않음.

v2 재검증 (5/19): 최신 PNG 재다운로드 후 우측 swim-lane (저장/복구/진입) 정밀 분석 결과:

  • LearningStateRepo.createInitial(...) → Local 저장 → Retry 라벨이 붙은 회귀 화살표 존재
  • 초기화 결과? 다이아몬드 분기 → 성공: GlobalLangState 갱신 / 실패: 지연 처리 / Retry 빨간 박스
  • 다른 다이어그램(d5/d6/d7/d8)과 동일한 Rollback + Retry + pending 패턴 일관성 확인됨

결론: 1차 검증 시 보조강사가 우측 swim-lane의 Retry 분기를 누락 인식. 사실관계 오류 → 철회.

⚠️ 검토 권장 (v2 재검증)

A. d4 AI Chat — mid-session disconnect 처리 모호 v2 하향 (← 1차 Critical #2)

v2 재검증 결과: step 10 "CHAT-006 종료 및 재진입 후 누적 turn 표시" 박스 + legend "Firebase Live API 세션은 짧게 짧게 ... LiveSession을 새로 생성할 수도 있음" 기술로 계획된 재진입(planned re-entry)은 시각화됨.

그러나 예기치 못한 LiveSession 끊김 → partial transcript 폐기 vs 임시저장 결정은 여전히 명시되지 않음. Gemini Live의 비정형 disconnect를 어떻게 처리할지가 PTT 입력 한가운데 터지면 사용자 체감에 직접 영향.

💡 권장: legend 1줄로 충분 — "예기치 못한 끊김 시 partial transcript는 폐기 (or 임시저장) → 사용자 재PTT 안내" 정책 명시. 별도 분기 박스까지는 불필요.

B. d5 Correction — 신규 사용자 Session Memory 빈 케이스 모호

"SessionSummary available?" 분기 → No → Empty 경로는 있으나, AI Chat을 한 번도 안 한 신규 사용자가 Correction 진입 시 recentFullContext = null 케이스 화면 동작이 다이어그램에서 식별 안 됨.

💡 권장: "교정할 발화가 아직 없습니다 — AI Chat 권유" 빈 상태 박스 1개 추가. 코드 작성 단계에서 자연 보완 가능 항목이므로 P1 → P2 수준.

C. 전 다이어그램 공통 — 권한 / 오프라인 시작 경로 누락 (원 #6)

d1 온보딩에 "Google 로그인 → Auth / Error" 분기는 있으나, 앱 최초 실행 시 오프라인 상태에서 Firestore 접근 실패 케이스가 모든 다이어그램에서 빠져있음.

김명준이 5/17 겪었던 (default) DB 이름 사고처럼 환경 / 네트워크 문제는 다이어그램 표준 분기로 박아두면 디버깅 시간 단축.

💡 권장: 각 Entry 박스 옆에 "오프라인 시작 → 캐시 사용 또는 차단 화면" 1줄 추가.

5. d1-B 저장 책임 매트릭스 — 정합성 충돌 의심 철회

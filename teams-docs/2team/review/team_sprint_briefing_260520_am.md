## 2. 블록 1 — 데모시나리오 종료 기준

### 결정 원칙

- 두 안(스프린트 기준 vs 플로우 기준) **모두 가능**. 팀이 합의한 방향이 정답
- **합의 자체에 시간 끌지 말 것**

### 보조강사 추천: 플로우 기준 ⭐

**근거 2가지**:

1. **플로우 문서에 이미 성공 기준·목표치가 작성되어 있음** → 모듈 단위 체크 용이
2. **플로우 단위 테스트가 화면 동작 흐름 검증에 자연스러움** → end-user 관점 데모와 정합

### 산업 표준 정합성 (어제 브리프 §3 검증)

| 출처 | 핵심 |
|---|---|
| Atlassian Sprint Demo | 데모는 user journey 워크스루 형태가 권장 |
| UAT Best Practices (Quellit) | 사용자 스토리 기반 end-user 관점 시나리오 |
| 오픈소스컨설팅 애자일 가이드(4) | 데모 가능한 시나리오로 표현할 것이 스토리 작성의 중요 기준 |

→ "플로우 기준 추천"은 산업 표준과 정확히 일치. 그대로 발화 가능.

## 3. 블록 2 — "플로우가 스프린트 안에 안 끝나면?" 조정 옵션

### 권장 발화 순서

#### (1) 옵션 A (1순위) — 플로우 자체를 더 작게 쪼개기

> "플로우 자체가 너무 크게 산정되었다면 **플로우를 더 작게 쪼개는 방향**이 1순위입니다."

#### (2) anti-pattern 한 줄 차단 — **필수**

플로우 분할만 권고하면 *"그럼 기간을 늘리면 안 되나요?"* 반문 가능. 한 줄로 차단:

> "**스프린트 기간을 플로우에 맞춰 늘리는 건 일반적으로 권장되지 않는 패턴(anti-pattern)이라**, 플로우를 더 작게 쪼개는 방향이 1순위입니다."

#### (3) 옵션 B (예외) — 본질적으로 못 쪼개지면 칸반 전환

> "만약 플로우가 본질적으로 짧게 못 쪼개진다면, 그땐 차라리 **고정 스프린트가 아닌 칸반(flow-based)으로 전환**하는 게 맞습니다. 다만 2팀은 이미 스프린트 cadence가 안정화되어 있으니 굳이 필요 없고, **옵션 A(쪼개기)가 1순위**입니다."

→ Kanban 전환은 *"만약 ~ 라면 그땐 차라리"* 예외 옵션으로만 언급 (자세한 이유는 §7 참조)

### 산업 표준 근거 (어제 브리프 §3)

| 출처 | 핵심 인용 |
|---|---|
| Scrum.org "27 Sprint Anti-Patterns" | *"스프린트 길이를 며칠 연장해 Sprint Goal을 맞추는 것은 agile이 아니라 **책 조작(cooking the books)**"* |
| Age-of-Product | *"이해관계자 참여(stakeholder inclusion)를 망가뜨리고, 스크럼 이벤트의 정상 cadence를 무너뜨린다"* |
| Agile Alliance | *"사용자 스토리는 반드시 하나의 스프린트에 들어가도록 설계되어야 한다. 안 들어가면 **쪼개라(split), 기간을 늘리지 마라**"* |

## 4. 블록 3 — 업무 분담 (학습 단계 단서 필수)

### 핵심 메시지

- **데모시나리오**: 한 명 전담 ❌ → 스프린트당 플로우 2~4개를 **각자 1개씩 맡고 서로 논의**
- **문서 작업도 동일**: **한 명에게 몰린 업무 구조** → 분담 + 상호 리뷰

### 학습 단계 단서 — **누락 시 오해 위험 ⚠️**

이 단서 없이 *"같이 진행하자"* 만 가면 *"그럼 실무에서도 분담이 정답인가?"* 반문 발생.

> "실무에서는 단일 담당이 책임 명확화·속도 면에서 더 효율적인 경우도 많습니다. **다만 지금은 부트캠프 학습 단계**라, 모두가 한 번씩은 데모시나리오·문서 작성을 경험해야 실무에서 혼자 작성할 수 있는 역량이 생깁니다. 그래서 **이번 프로젝트 시점에는** 분담 + 상호 리뷰를 권장합니다."

### 산업 자료 검증 (어제 브리프 §3 D·E)

| 입장 | 출처 |
|---|---|
| 단일 담당 옹호 | Quora/Scrum.org: 책임 명확화·핸드오프 최소화·속도 |
| 분담·페어 옹호 | Atlassian/Quellit: pairing developers with testers, 짝 단위 한 시나리오 함께 작성 |

## 5. 블록 4 — 차주 스프린트 문서 범위 + 공동 소유

### 핵심 메시지

#### (1) 최소 필요분만 빠르게 합의

> "당장 차주 스프린트에 **필요한 문서만** 골라 빠르게 합의하고, 합의 후 즉시 분담합시다."

#### (2) 공동 소유(collective ownership) ⭐

> "이렇게 진행해도 빠지는 부분은 무조건 생깁니다. 다만 빠진 부분을 발견했을 때 **담당자에게 다시 작업해달라고 요청하지 말고, 발견한 사람이 직접 수정하거나 같이 다듬는 방향**으로 가는 게 좋습니다. 누구나 수정할 수 있어야 학습 단계에서 시야가 넓어집니다."

→ 이게 사용자 4안 중 **가장 강한 메시지**. *"공동 소유"* 단어 한 번 얹어주면 정착력 ↑.

### 정원화 호명 회피 ⚠️

직접 *"정원화 훈련생이 혼자 맡고 있으니 분담"* 으로 가지 말고, **일반론 → 자연스러운 분담 합의 유도**:

> "**한 명에게 몰린 업무 구조**에서는 빠진 부분도 그 사람만 책임지게 되기 쉽습니다. 그래서 공동 소유 방향으로 가는 게..."

근거: 5/16 정원화 1:1 *"본인 능력에 강박적 align"* 패턴 + 5/19 태환 1:1 *"원하님이 진행하면서 구체화되어서 나누기 힘들어요"* 구조적 원인.

## 6. [심화] 왜 스프린트 기간 가변 조정이 anti-pattern인가 — 5가지 이유

> 블록 2에서 anti-pattern 한 줄 언급할 때 *"왜요?"* 질문 받을 대비. 5가지 중 2팀에 가장 강한 ①·⑤만 압축 답변 권장.

### ① 타임박스 자체가 강제 학습 장치

스프린트 = 고정 타임박스. 무너지면:

- *"무엇이 너무 컸는가"* 학습 못 함 — "기간 안에 안 들어갔다"는 사실 자체가 **추정 정확도·플로우 크기·DoD 모호함**을 드러내는 신호
- 기간을 늘려서 맞추면 그 신호가 사라짐
- Scrum.org: *"Sprint Goal을 맞추기 위해 며칠 더 끄는 것은 agile이 아니라 책 조작(cooking the books)"*

→ **회고에서 *"왜 못 끝냈나?"* 라는 가장 중요한 질문이 사라진다.**

### ② Velocity 측정 불가 → 다음 계획 불가

- 스프린트 길이가 매번 달라지면 velocity 측정 기반이 사라짐
- *"우리가 2주에 몇 포인트 처리 가능한가"* 답이 안 나옴 → 계속 즉흥 운영
- 2팀의 5/18 v2 peak 83.3% + Sprint 2 완전 이행 사이클이 무너짐

### ③ 이해관계자 신뢰·참여 메커니즘 파괴

- Age-of-Product: *"이해관계자 참여를 망가뜨리고, 스크럼 이벤트의 정상 cadence를 무너뜨린다"*
- 데모일·리뷰일이 고정이어야 강사·다른 팀이 참여 가능
- 가변 조정 = *"이번엔 언제 끝나는지 모름"* → 외부 일정 조율 불가
- 신뢰 = 약속한 날짜에 약속한 만큼 본다는 예측 가능성에서 나옴

### ④ Parkinson's Law — 일은 주어진 시간만큼 늘어난다

- 기간 연장 한 번 허용 → 다음 스프린트도 "조금만 더" → 끝없이 늘어남
- 타임박스 압박이 **쪼개기·우선순위 조정·MVP 단순화** 라는 정상 결정을 유도
- 압박이 사라지면 *"기능을 다 넣고 싶다"* 욕구가 이김 → 어제 태환에게 가르친 *"본인 최선 상태 가정 금지 + 1.5배 보정"* 원칙이 작동할 자리 사라짐

### ⑤ R-burn 가속 — 2팀 코호트 직접 리스크 ⚠️

**2팀에서 가장 무거운 이유**:

- 기간을 늘리면 *"그 며칠 더 일하면 끝낼 수 있다"* → **주말·야간 작업으로 흘러감**
- 어제 사용자가 태환에게 가르친 *"주말 일정 금지"* 가 정면으로 깨짐
- 현재 2팀 상태: 정원화 R-burn 자율 추가 + 김명준 R-burn-2 재발 + 5/19 김명준 새벽 3건 / 정원화 자정 작업 패턴
- **기간 가변 허용 = R-burn 가속기**

### 발화용 압축 한 줄 (블록 2 보강)

> "스프린트 기간을 늘리는 건 일반적으로 권장되지 않는 패턴입니다. 이유는 두 가지인데, 첫째 **회고에서 *왜 못 끝냈는가* 라는 학습 기회가 사라지고**, 둘째 **기간을 늘리는 만큼 작업이 주말·야간으로 흘러가서 결국 팀이 지쳐버립니다**. 그래서 플로우 자체를 더 작게 쪼개는 방향이 1순위입니다."

→ "학습 기회 + R-burn 예방" 두 축이 2팀이 가장 잘 받는 톤.

## 7. [심화] 애자일 vs 스크럼 vs 칸반 — 개념 정리

> 블록 2 옵션 B(칸반 전환)에서 *"그럼 애자일 버리는 거 아니냐?"* 질문 받을 대비.

### 핵심: 애자일 ≠ 스크럼

```
                  Agile Manifesto (2001)
                  (4가지 가치 · 12원칙)
                          │
        ┌──────────┬──────┴──────┬────────────┐
      Scrum     Kanban         XP          Lean / Crystal / ...
   (타임박스)   (흐름)      (엔지니어링)
```

- **애자일 = 가치·원칙의 집합** (개인과 상호작용 / 작동하는 SW / 고객 협력 / 변화 수용)
- **스크럼·칸반·XP = 그 가치를 구현하는 서로 다른 프레임워크**
- "스프린트를 버리고 칸반으로" = **애자일을 버리는 게 아니라, 애자일 안에서 프레임만 바꾸는 것**

### Scrum vs Kanban — 핵심 차이

| 축 | Scrum | Kanban |
|---|---|---|
| 시간 단위 | **고정 스프린트** (1~4주) | **연속적 흐름** (시간 단위 없음) |
| 계획 | 스프린트 단위 일괄 계획 | 작업 들어올 때마다 풀(pull) |
| 핵심 제약 | 타임박스 | **WIP 제한** (동시 진행 작업 수) |
| 측정 지표 | Velocity (스프린트당 처리 포인트) | Lead time / Cycle time (작업 1건 처리 시간) |
| 역할 | PO / SM / Dev 강제 | 역할 강제 X |
| 의식 | Planning/Daily/Review/Retro 강제 | 의식 강제 X |
| 적합 상황 | 요구사항이 스프린트 단위로 묶일 때 | 작업 크기·우선순위가 **수시로 바뀔 때** |

### 왜 "스프린트 늘리기"는 나쁘고 "Kanban 전환"은 괜찮은가?

같은 "기간 안 맞음" 상황인데 차이가 큰 이유:

- **스프린트 늘리기** = 스크럼 프레임을 **고장 난 상태로 계속 쓰는 것** → cadence·velocity·회고 다 무너지는데도 "우리 스크럼 하고 있어요" 라고 자기기만
- **Kanban 전환** = 스크럼이 안 맞는다는 사실을 인정하고 **WIP 제한·Lead time 측정이라는 다른 정상 프레임으로 갈아타는 것** → 여전히 애자일 원칙(짧은 피드백 루프, 작동하는 SW, 변화 수용) 다 살아있음

→ 핵심: *"프레임 깨고 그대로 가는 게 가장 나쁨"*. **프레임을 바꾸거나 프레임을 지키거나 둘 중 하나**.

### 중간 옵션 — Scrumban (참고용)

스크럼과 칸반 중간 형태. 스프린트는 유지하되 백로그를 칸반처럼 풀 시스템으로 운영. 한쪽으로 완전히 가기 부담스러울 때 쓰는 하이브리드. **2팀은 굳이 필요 없음**.

### 3팀의 "워터폴+칸반" 은 다른 케이스

3팀은 **애자일 자체를 안 함** (워터폴 확정):

- **워터폴 단계** (요구 → 설계 → 구현 → 테스트 → 배포): 비-애자일. 단계 순차 진행
- **각 단계 내 실행**: 칸반으로 운영 (WIP 제한, 흐름)
- = **워터폴 + 칸반 하이브리드** (강사 5/15 PM 공식 인정)

→ 3팀은 처음부터 *"애자일 안 한다"* 가 결정 사항이라, 2팀의 *"스크럼 cadence가 깨지면 칸반으로 전환"* 시나리오와는 출발점이 다름. **3팀은 워터폴 단계 안에서 칸반을 도구로 쓰는 것**.

### 2팀 적용 매트릭스

| 상황 | 추천 프레임 | 이유 |
|---|---|---|
| 지금 2팀 | **스크럼 유지** ⭐ | 5/18 v2 peak 83.3% + Sprint 2 완전 이행 = cadence 안정. 굳이 바꿀 이유 없음 |
| 플로우가 본질적으로 못 쪼개지면 | **Kanban 전환 검토** (예외) | 스프린트 깨고 그대로 가느니 차라리 프레임 바꿈 |
| 절대 피할 것 | **스프린트 기간 가변 조정** ❌ | 가장 나쁜 옵션 — 모든 장점 다 잃음 |

### 발화용 한 줄

> "**애자일(Agile) = 가치·원칙** / **스크럼·칸반 = 그 안의 프레임워크**. 스프린트를 버리고 칸반으로 가는 것은 애자일을 버리는 게 아니라, **애자일 안에서 더 적합한 프레임으로 갈아타는 것**. 가장 나쁜 건 *'스크럼인 척하면서 타임박스만 늘리기'* 입니다."

- 정원화 훈련생 직접 호명 회피

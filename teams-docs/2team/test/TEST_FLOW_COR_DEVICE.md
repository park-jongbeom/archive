## 2. 디바이스 단독 오류·엣지 케이스 유발 시나리오

각 시나리오는 (a) 정상 TC 와 (b) 오류 유발 변형 으로 쌍을 이룬다. **모두 디바이스 UI 동작 + 네트워크 토글 + 화면 이탈 + 회전 등만 사용** 한다.

### 2-1. 교정 진입 단계 (TC-COR-01 / 02 / 03)

| # | 유발 방법 | 기대되는 오류·엣지 | 검증 포인트 |
| --- | --- | --- | --- |
| E-01 | Chat 에서 **1초 미만 발화 1턴** 만 하고 Dashboard 복귀 → 교정 카드 클릭 | "적어도 1개의 의미있는 사용자 턴이 포함되어야 한다" 류 안내 또는 Empty 상태 | 안내가 사용자에게 명확한가, Retry/Chat 이동 CTA 가 있는가 |
| E-02 | Chat 에서 **침묵 상태로 PTT press → 즉시 release** 1턴 진행 → 교정 카드 클릭 | partial only / final 없음 → 교정 대기 카드 자체가 비활성이어야 함 | 카드 활성 여부, 클릭 시 차단 메시지 |
| E-03 | Chat 에서 USER final 1건 저장 후 **즉시(1초 내) Dashboard 복귀 → 교정 카드 클릭** | SessionSummary 동기화 race → Empty 또는 잠시 안내 후 정상 | race 처리 정상성 |
| E-04 | Dashboard 에서 교정 카드 클릭 **직전 비행기 모드 ON** → 클릭 | AI 교정 호출 timeout → Error + Retry | 5단계 로딩 끝까지 도는지(15초 보장이 문제 되는지), Error 문구 적절성 |
| E-05 | Dashboard → 교정 카드 **이중 탭** | 중복 진입 차단 (COR-001 AC8) | 진입 1회만, 로딩이 두 번 안 뜸 |
| E-06 | 학습 언어 selector 변경 직후 즉시 교정 카드 진입 | 언어 컨텍스트 race | 새 언어 기준으로 일관되게 동작 |

## 3. 알려진 결함 영역 (원본 시트 테스터 코멘트 기반)

원본 시트의 비고 칸에서 다음 패턴이 반복 관측됨. 디바이스 검증 시 우선 주시.

| 영역 | 관측 내용 | 디바이스 검증 시 주의 |
| --- | --- | --- |
| 저장 실패 메시지 | "적어도 1개의 의미있는 사용자 턴이 포함되어야 한다" 안내가 PendingSync/SaveFail/TopicTitleEmpty 등 다양한 시나리오에서 노출 | 사용자가 원인을 구분 불가. 짧은 발화 외에도 다른 케이스에 같은 메시지 노출되는지 추적 (E-01 + E-19 + 저장 후 재진입) |
| 저장 중 진행 표시 모호 | TC-COR-14 비고: "재시도 시 저장 중인지 알기 어려움" | E-18 (빠른 멀티 탭) 과 E-21 (회전) 에서 진행 표시가 끊기는지 확인 |
| Error 진입 시 불필요한 로딩 | TC-COR-04 비고: "Error 상태는 guard 초기 조건으로 해서 불필요한 로딩 화면 줄이면 좋겠다" | E-04 (비행기 모드 ON 후 진입) 에서 15초 로딩 끝까지 도는지 확인. 사용자가 너무 오래 기다리는 인상 받는지 |

## 4. 디바이스 검증 권장 순서

1. **happy path 1회** — TC-COR-01 → 02 → 03 → 11 → 15 → 16 → 18 → 22 → 24 정상 통과 확인
2. **로딩 영역 오류 유발** — E-04, E-07, E-09, E-10 (재현율 높음)
3. **저장 영역 오류 유발** — E-18, E-19, E-21 (결함 누적 빈도 높음)
4. **상태 일관성** — E-22, E-24, E-27 (이중 처리·중복 토스트)
5. **언어/세션 컨텍스트** — E-06, E-17, E-25 (관리 누락 자주 발생 지점)

## 5. 검증 결과 기록 권장 양식

원본 TC 표의 비고 칸과 동일 형식으로 기록한다.

```
[E-NN] <유발 방법 한 줄>
관측: <화면·메시지>
정상 여부: P / F / 세모
재현율: N/M (예: 2/3)
사이드 노트: <필요 시>
```

판정은 원본 시트의 "판정 기준" 섹션을 그대로 따른다.

- 사용자가 다음 행동을 이해할 수 있으면 Empty/Error/Retry 도 Pass
- 저장 실패/Retry phase 에서 Dashboard 이동 발생 시 Fail
- pending sync/compression 이 사용자에게 실패로 보이면 Fail

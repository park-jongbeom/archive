## 2. 잘한 점 (그대로 유지하세요)

1. **워치 연동을 핵심 가치 그 자체로 설계**
   다른 많은 팀이 *"워치를 어떻게 끼워 넣지?"* 로 시작하는 반면, Scoffee는 *"워치 알림이 핵심 가치"* 라는 명확한 의지를 한 줄 소개에서부터 보여줍니다. 부트캠프 6기 워치 연동 요건을 단순 충족이 아니라 **제품 정체성으로 흡수**한 매우 좋은 설계입니다.

2. **차별점 한 줄 명확** ([가이드 § 4-4](../../../final-project/docs/애자일/01_애자일_팀프로젝트_가이드.md))
   *"일반적인 '오후 2시 이후 금지' 가 아니라 내 수면 데이터를 학습한 AI(Gemini)가 나만의 카페인 컷오프 시간을 알려주는 앱"* — 경쟁 제품 대비 명확한 한 줄. 발표/IR 어디서든 그대로 사용 가능.

3. **제약사항 자기 인식**
   *"WearOS 레퍼런스 부족"* — 비전공자/주니어 팀에서 자기 한계를 정직하게 명시하는 건 드뭅니다. 1주차에 *"이 부분은 mock으로 시작하자"* 같은 의사결정이 자연스럽게 가능해집니다.

4. **Flow A/B 두 줄기로 분리**
   FLOW-A (입력 → 잔류량 → 워치) / FLOW-B (수면 → 리포트). 가이드 § 5-2-1 권장 *"1~2개"* 정합. Week 2~3 = A 완성, Week 4~5 = B 완성으로 6주 운영 리듬 그대로 들어맞음.

5. **성공 기준 관찰 가능**
   *"FLOW 무중단 + 로딩/에러/빈 1회 이상 포함"* — 가이드 § 5-2 *"Loading/Error/Empty 최소 1회"* 그대로 반영. 데모 평가 기준이 명확.

## 3. 짚어드릴 점 (v2 — 톤 완화)

### 3-1. 🟢 Must 7개 도전 가능 — 단 2가지 위험 분산 권장

**v1 권장 (강도 ↑)**: Must 7개 → 4~5개로 압축
**v2 권장 (강도 ↓)**: Must 7개 도전 그대로 가되, 학습 곡선 위험을 **mock-first로 분산**

**이유**: 4명 중 3명이 프로젝트 경험 보유 → 표준 기능 7개 동시 진행 가능. 다만 다음 2가지는 학습 곡선 + 외부 의존 위험이 있어 1주차에 분산 전략 합의 권장.

#### 3-1-A. WearOS 알림 3종 — 단계적 전개

**현재 Brief**: Must = "WearOS 알림 3종 (섭취 직후 / 컷오프 도달 / 취침 전)"

**위험**: WearOS Notification + Tile + Health Services 통합은 러닝 커브 + 디바이스 페어링 디버깅 시간 소요.

**권장 전개**:
| 알림 | 주차 | 분류 |
|---|---|---|
| 섭취 직후 알림 (가장 단순) | Week 2 데모에 포함 | Must |
| 컷오프 도달 알림 | Week 3~4 | Must (Week 4까지 완성 목표) |
| 취침 전 알림 | Week 4~5 | Must (또는 Should로 강등 옵션) |

→ Brief 변경 없이 **내부 우선순위만 합의**해도 충분.

#### 3-1-B. Health Services 수면 데이터 연동 — mock-first

**현재 Brief**: Must = "수면 데이터 연동 (Health Services)"

**위험**: Samsung Health 권한 + 데이터 동기화 + 사용자 권한 거부 시나리오 — 1주차 시작부터 권한 다이얼로그가 막히면 다른 작업이 멈춤.

**권장**:
- **Week 1~2**: mock 수면 데이터(`mock_sleep_data.json`)로 잔류량-수면 상관관계 계산 로직 구현
- **Week 3 시작 시**: 실제 Health Services API로 교체 (별도 Issue로 분리)
- **Decision Log 1줄**: *"수면 데이터는 Week 1~2 mock, Week 3 실연동"*

#### 3-1-C. Gemini API 컷오프 산출 — mock-first

**현재 Brief**: Must = "개인화 카페인 컷오프 시간 산출 (Gemini API)"

**위험**: Gemini API 비용 / 응답 속도 / 프롬프트 안정성 — 본인들도 제약사항에 명시.

**권장**:
- **Week 1~2**: 정해진 공식 (반감기 기반 컷오프) 으로 mock 응답
- **Week 3~4**: Gemini API 통합 + 프롬프트 튜닝
- **Decision Log 1줄**: *"Gemini는 Week 3 이후 통합, 그 전엔 공식 기반 mock"*

**3개 mock-first 합의 후 Brief 변경 사항**: 없음 (Must 그대로 유지). 단 [Decision Log](../../../final-project/docs/애자일/example2/1팀/06_decision_log.md) 에 3줄 결정 기록만 추가.

### 3-2. 🟡 팀 구성 고려 — 작업 분배 + 역할 보조 (신규)

**상황**: 팀장 송성호님이 비전공자 + PO 역할. 가이드 § 11-1 — *"PO는 우선순위 결정, Flow 변경 관리"*. 비전공자가 PO를 맡으면 **기술 트레이드오프 판단**이 어려움.

**가이드 § 11**: *"역할을 전담하지 못해도 괜찮습니다. 책임만 끊기지 않게"*

**권장 — 역할 보조 구조** (1주차 30분 회의):

| 책임 | 담당 |
|---|---|
| **사용자 관점 / 제품 우선순위** | 송성호님 (PO 본연 책임) — 비전공자 시각 = 사용자 시각 강점 |
| **기술 트레이드오프 / 아키텍처 판단** | 손지희님 (부팀장, 전공자) — 비공식 테크 리드 |
| **개발 작업** | 전원 (PO 포함) |
| **리뷰** | 전공자 3명 로테이션 |

이렇게 *"PO 의사결정에 부팀장이 기술 자문"* 구조를 1주차에 명시하면, 송성호님이 *"이거 가능해요?"* 같은 질문을 부팀장에게 던지고 PO 결정을 내릴 수 있음. 결정 부담 분산.

#### 3-2-A. 송성호님 학습 곡선 배려 — 영역 추천

비전공자가 "쉬운 작업만"을 하면 6주 후 학습 격차가 더 커집니다. **쉬운 영역에서 시작 + 페어 프로그래밍으로 점진적 확장** 권장.

| 시작 영역 (Week 1~2) | 학습 곡선 낮음 | 추천 페어 |
|---|---|---|
| ✅ 카페인 입력 화면 (form + validation) | 표준 Compose 패턴 | 손지희님 페어 |
| ✅ 온보딩 화면 (3 step) | 표준 패턴 | 이제이님 페어 |
| ✅ 카페인 음료 마스터 데이터 (DB seed) | CRUD 표준 | 임정섭님 페어 |

| 점진 확장 (Week 3~4) | 페어 프로그래밍 권장 |
|---|---|
| 잔류량 계산 UI 표시 | 알고리즘은 전공자, UI 통합은 송성호님 |
| 워치 알림 발신 트리거 | WearOS 셋업은 전공자, 알림 메시지 작성은 송성호님 |

| Week 5~6 | 단독 작업 가능 영역 |
|---|---|
| 주간 리포트 차트 UI | (이전 작업 누적 학습 후) |

**원칙**: 송성호님이 "PR 1개 직접 머지"를 매주 1번 이상 달성하면 OK. 가이드 § 4 *"Week 1: PR 1개 이상"* 정합.

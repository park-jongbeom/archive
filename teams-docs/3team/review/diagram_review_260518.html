1. d1 ① DM 메시지 ↔ d5 DM 도메인 — "온라인 / 다른 화면" 분기 두 다이어그램에서 충돌 v2 유지

  • d1 상단 sequence: "분기 2 (온라인 / 다른 화면) → Notifications 생성 → 라이브 리스너로 즉시 알림 수신" — 푸시 없음.
  • d5 하단 DM flowchart: ServerSaveInactive 단일 분기에서 "Notifications 생성 + 오프라인 푸시 발송" 묶음 — 온 / 오프 구분 없이 푸시 발송.

같은 시나리오를 두 다이어그램이 다르게 그림. 온라인 - 다른화면 사용자에게 FCM을 쏘는지 안 쏘는지 합의 필수.

💡 권장: d5의 ServerCheckOpponent를 3분기로 쪼개기 (대화방 내 / 온라인 - 다른화면 / 오프라인) — d1 sequence와 동일 구조로.

2. d1 ③ 음성 메시지 — "수신자 본인이 자기 문서 삭제" 권한 모순 v2 유지

v2 재검증 (systemflows L70): Receiver->>FS: 재생 완료 후 해당 문서 삭제 (휘발성 유지) 그대로.

"재생 완료 후 Receiver ->> FS: 해당 문서 삭제 (휘발성 유지)" 라고 클라이언트가 직접 Firestore에 delete 호출.

그런데 강사 5/18 PM 피드백: "모든 쓰기(Write) 작업과 타인 정보의 접근은 '서버(Cloud Functions)'를 통해서만 수행한다"

VoiceMessages/{receiver_uid} 문서는 수신자 본인 소유라 보안 규칙상 self-delete 허용은 가능하지만, "쓰기는 무조건 서버" 원칙과 충돌하는 예외.

💡 권장: 둘 중 하나 결정:

  • (a) 예외 명시 — "본인 소유 문서의 read / delete만 클라이언트 허용" 룰을 다이어그램 주석에 박기
  • (b) deleteVoiceMessage Callable 함수 경유로 통일

3. d1 ③ 음성 메시지 — 오프라인 트리거 조건 updated_at 체크가 모호 v2 유지

v2 재검증 (systemflows L72): FS-->>CF: 트리거 작동 (updated_at 체크) 그대로. d1 ① DMs 패턴(Live/{uid} 조회 후 분기)과 여전히 일관성 안 맞음.

"수신자 오프라인 → FS->>CF: 트리거 작동 (updated_at 체크)"

Firestore 트리거는 문서 생성 / 수정 즉시 발화하는데, "오프라인일 때만 FCM 보내고 온라인이면 안 보낸다"는 판정을 트리거 시점에 어떻게 하나?

  • 옵션 A: 트리거가 무조건 발화 → CF가 Live/{uid} 조회해서 분기 (d1 ① DMs 패턴과 동일)
  • 옵션 B: 클라이언트가 N초 내 삭제 안 하면 onUpdate 트리거 → 위험 (race condition, 재생 끝나기 전 삭제 vs FCM)

💡 권장: d1 ① DMs와 동일하게 트리거 즉시 Live/{uid} 조회 분기로 통일. "updated_at 체크" 표현 자체가 혼란.

⚠️ 검토 권장

4. d2 계정인증 — 소셜 로그인 후 닉네임 충돌 처리 누락 v2 유지 (약간 완화)

v2 재검증 (systemflows L133): BatchCommit에 "(오류 시 전체 자동 취소)" 문구 신규 추가됨. 트랜잭션 원자성 의도는 명시되었으나 닉네임 unique 충돌 자체의 사전 회피 분기(suffix 부여 등)는 여전히 없음.

이메일 가입 → UpdateRealNickname으로 "진짜 닉네임"으로 즉시 덮어쓰기. 소셜 가입 → ParseNickname이 만든 임시 닉네임 그대로 두고 FetchToken으로 넘어감.

그런데 임시 닉네임(이메일 ID 또는 소셜 프로필명)이 이미 다른 유저가 쓰고 있으면? 닉네임 unique 제약이 있다면 BatchCommit 자체가 실패.

💡 권장: ParseNickname 다음에 "중복 확인 → 충돌 시 suffix 부여" 분기 추가, 또는 "닉네임 unique 제약 없다" 결정 주석.

5. d2 계정인증 — AuthCheck "로그인 정보 확인됨" 경로의 데이터 동기화 빈틈 v2 유지

v2 재검증 (systemflows L146): AuthCheck -- "로그인 정보 확인됨 (기존 회원)" --> FetchToken 그대로. 기존 유저 프로필 / Live 로딩 단계는 여전히 누락.

기존 회원 로그인 시 AuthCheck → FetchToken → UpdateFCM → ... 으로 바로 점프.

그런데 앱 재설치 / 기기 변경한 기존 회원은 Users / Live 문서가 이미 존재하지만 로컬 캐시가 비어있음 → 다이어그램 어디에도 "기존 유저 프로필 / Live 로딩" 단계가 없음.

💡 권장: AuthCheck "확인됨" → LoadExistingProfile (Users + Live 조회) → FetchToken 순서로 한 노드 추가.

6. d3 홈화면 — Foreground 모드의 BgFriendSync 누락 v2 유지

v2 재검증 (systemflows L184): FgSync["화면 활성화 시에만 데이터 동기화 : onResume 시작 / onPause 중지"] 단일 노드. "데이터 동기화"에 친구 위치가 포함되는지 여전히 모호.

StartBgService 분기는 BgUserSync + BgFriendSync + WearableListener 3종 모두 시작.

StartFgMode 분기는 FgSync 하나만 — 친구 위치 리스너는? 친구 마커가 지도에 안 뜨는 모드?

명시적으로 "포그라운드에선 친구 위치 끔" 결정이면 OK, 아니면 빠진 것.

💡 권장: FgSync에서도 친구 리스너는 onResume 시 작동한다는 표기 추가, 또는 "포그라운드는 내 위치만" 결정을 다이어그램 주석으로 명시.

7. d3 홈화면 — 무전(PTT) 권한 거부 후 흐름 막다른 골목 v2 유지

v2 재검증 (systemflows L215): MicPermRequest -- "거부됨" --> ClosePopup 그대로. 설정 이동 안내 노드 여전히 없음.

MicPermRequest -- "거부됨" --> ClosePopup — 권한 거부 시 그냥 팝업 닫음.

사용자가 "왜 거부했나? 다시 누르면 또 거부 다이얼로그?" → 이미 거부한 권한은 OS가 다이얼로그 안 띄움. 설정 이동 유도 필요.

💡 권장: PermCheck 최상단의 ShowEmptyState → GoToSettings 패턴과 동일하게, 마이크 거부도 "설정에서 권한 허용 필요" 안내 → 설정 이동 노드 추가.

8. d4 인앱 알림 — FRIEND_ACCEPTED vs FRIEND_REQUEST 구분 없음 v2 유지

v2 재검증 (systemflows L283): UserAction -- "친구 수락 알림 클릭" --> CallMarkAsReadFriend 단일 케이스 그대로.

친구 알림 흐름은 UserAction -- "친구 수락 알림 클릭" 단일 케이스만 그려짐.

그런데 d1 ② 친구 요청 sequence에서는 요청 송신자가 받는 알림 + 수락한 후 송신자가 받는 알림 두 종류가 모두 생길 수 있음.

요청 받은 사람이 누르면 → 친구 목록? 요청 대기 목록? 결정 필요.

💡 권장: FRIEND_REQUEST(받음, 수락 대기) vs FRIEND_ACCEPTED(보냈는데 상대 수락) 두 케이스로 분기 추가. 진입 화면도 다름.

9. d4 인앱 알림 — PlayAudio 후 휘발성 원칙 충돌 v2 유지

v2 재검증 (systemflows L270-276): CheckValidity는 "유효기간"만 체크 · AudioEnd: "재생 종료 (목록에서는 지워지지 않고 유지)" 그대로. VoiceMessages 원본 삭제 후 Notifications 재생 버튼 재클릭 시나리오는 여전히 모호.

d1 ③에서 "재생 완료 시 문서 삭제 (휘발성 유지)" → VoiceMessages 자체는 삭제.

그런데 d4는 AudioEnd: "목록에서는 지워지지 않고 유지" → Notifications 문서는 유지.

두 문서가 다른 컬렉션이라 양립 가능하지만 — VoiceMessages 삭제됐는데 Notification에 "재생 버튼"이 다시 보이면? ValidState → PlayAudio 재시도 시 원본 없음 에러.

💡 권장: CheckValidity에 "원본 VoiceMessages 존재 여부" 체크도 포함. 또는 재생 후 Notifications 항목에 played: true 플래그 표시.

10. d5 DM 도메인 — unread_counts 초기값 누락 명시 v2 유지

v2 재검증 (systemflows L298-300): "1:1채팅방 개설 시 unread_counts 값 넣어주기" 노트는 mermaid 본문 위에 그대로 잔존. flowchart는 여전히 SubChatDirect(기존 방 입장)부터 시작 — 방 개설 trigger plot 없음.

다이어그램 위 주석에:

  • "1:1채팅방 개설 시 unread_counts 값 넣어주기"
  • "읽지 않은 알림 필드값 변경 처리 필요 → API 구현 필요 << 바로작업"

이걸 다이어그램 본문에는 안 넣고 위 주석으로만 둠. 방 개설 plot 자체가 다이어그램에 없음 = 채팅방 생성 트리거(=친구 추가? 첫 메시지?)가 어디서 들어오는지 불명.

💡 권장: DM flowchart 앞단에 "대화방 최초 생성 (양쪽 unread_counts: 0 초기화)" 노드 추가, 또는 별도 짧은 sequence 1장.

# 3팀 백그라운드 위치 처리 질의 — 피드백 노트

> **목적:** 3팀이 보조강사에게 제기한 "스냅샷 vs 주기적 갱신" 질의에 대한 타당성 평가와 구현 시 유의사항 전달.
> **방침:** 해결 방법은 본 문서에 포함하지 않음. **무엇을 조심해야 하는지**만 정리.

---

## 1. 질의 타당성 평가

### 결론: **질문 자체와 정우석님의 분석은 타당하다.**

- 위치 공유 앱의 본질적 트레이드오프(비용 vs 실시간성 vs 배터리)를 정확히 짚었음.
- Foreground Service 필요성, FusedLocationProvider, Priority 동적 전환, 거리 기반 필터링, Activity Recognition 도입 — 모두 표준 패턴과 일치.
- "스냅샷이 일회성이라 부적합"이라는 판단의 결론도 옳음.

다만 질의·답변 과정에서 **"스냅샷"이라는 단어가 두 가지 다른 기술을 동시에 가리키고 있어서**, 주강사·정우석 두 분 답변이 충돌하는 것처럼 보였을 뿐이다. 실제로는 서로 다른 레이어를 이야기하고 있었고, **두 분 모두 옳다.**

---

## 2. 반드시 인지해야 할 개념 구분 — "스냅샷"은 두 가지

| 구분 | 가리키는 기술 | 적용 레이어 |
|---|---|---|
| A | Firestore `addSnapshotListener` (gRPC 스트림 구독) | **READ** — 친구 위치를 내 앱으로 받기 |
| B | `FusedLocationProvider.getLastLocation()` (단발 조회) | **WRITE 측 입력** — 단말 GPS에서 내 위치 가져오기 |

- 주강사 답변(소켓 방식이라 좋다) = **A** 이야기.
- 정우석님 답변(일회성이라 부적합) = **B** 이야기.
- 두 개념은 **양자택일이 아니라 서로 다른 레이어**이며 결합되어야 한다.

---

## 3. READ 측(친구 위치 받아오기) 유의사항

- **백그라운드에서 SnapshotListener는 안정적으로 유지되지 않는다.**
  OS가 일정 시간 후 gRPC 소켓을 강제로 닫으며, 리스너는 먹통이 되거나 재연결을 반복하며 자원을 소모한다.
- **친구 위치가 실시간으로 보이려면 "친구 단말이 Firestore에 쓰고 있어야" 한다.**
  내 SnapshotListener는 친구 앱 상태를 알지 못한다. 친구 앱의 WRITE가 멈추면 내 화면에는 마지막 기록 위치만 남는다.
- **친구가 앱을 강제 종료하거나 위치 공유 OFF로 토글한 경우의 UX(예: 회색 마커, "마지막 위치"표기)를 명세 단계에서 정의하지 않으면, 구현 단계에서 모호한 화면이 나온다.**

---

## 4. WRITE 측(내 위치 쓰기) 유의사항

### 4-1. Foreground Service의 의미 정확히 이해 필요

- "앱이 종료되어도 동작한다"는 표현은 **부정확**하다.
  정확히는 **"앱 UI가 보이지 않는 백그라운드 상태에서도 프로세스가 살아있도록 OS와 맺는 계약"**이다.
- **상단 알림은 옵션이 아니라 계약 조건**이다. 사용자가 임의로 닫을 수 없도록 설계되며, 알림 없는 Foreground Service는 존재할 수 없다.
- Foreground Service라도 **다음 상황에서는 죽는다**:
  - 사용자가 설정 → 앱 정보 → **강제 종료**한 경우
  - **재부팅** 발생
  - 일부 OEM 단말에서 **태스크 스와이프**한 경우
  - 사용자가 권한을 회수한 경우

### 4-2. Android 14+ 정책 변화

- 서비스 타입 명시(`FOREGROUND_SERVICE_LOCATION` 등)가 **누락되면 앱 자체가 시작되지 않는다.**
- Play 스토어 심사 시 **"왜 백그라운드 위치가 필요한가"에 대한 소명**을 요구한다. 충분한 설명이 없으면 등록 거절 사례가 보고되고 있음.
- Android 15+에서는 일부 서비스 타입에 **실행 시간 제한(약 6시간)**이 도입되어, 장시간 유지 시 강제 종료 가능성이 있음.

### 4-3. 국내 단말(특히 갤럭시) 환경 리스크

- 삼성의 **"미사용 앱 절전"**, **"기기 케어 → 배터리"** 정책은 Foreground Service라도 백그라운드 프로세스를 적극적으로 종료한다.
- **사용자가 "배터리 최적화 예외"로 직접 등록하지 않으면**, 앱이 일정 시간 후 죽는 현상이 갤럭시에서 광범위하게 발생한다.
- 갤럭시 워치를 타깃으로 하는 본 앱의 특성상, **이 이슈는 데모 환경에서 100% 표면화될 가능성**이 있음.

### 4-4. Doze 모드

- 기기가 충전 중이 아니고 화면이 꺼진 상태로 일정 시간 지나면 OS가 네트워크 활동을 제한한다.
- Foreground Service 자체는 살아있어도, **Firestore 쓰기 요청이 지연되거나 묶여서 전송**될 수 있다.

## 5. Activity Recognition 유의사항

- **인식 지연 발생.** STILL → WALKING 전환 감지에 통상 30~60초가 소요된다. 사용자가 막 출발한 직후의 위치가 누락될 가능성이 있음.
- **오인식 빈번.** 차량 정차, 지하철 정지, 신호 대기 등에서 STILL로 잘못 분류되는 사례가 많다.
- **단독 신호로 사용 시 위험.** "Activity가 STILL이면 위치 수집 중단"처럼 단독 트리거로 두면, 위 두 함정이 그대로 사용자 경험 결함으로 이어진다.
- **권한 요구.** `ACTIVITY_RECOGNITION` 런타임 권한이 별도로 필요하며, 사용자가 거부한 경우의 fallback 동작이 명세에 없으면 구현 단계에서 막힌다.

---

## 6. Firestore 비용·구조 유의사항

- **단일 문서에 대한 쓰기는 초당 1회(1Hz)가 권장 상한.** 그 이상 주기로 명세를 잡으면 일시적으로 동작해도 부하 시 throttle/실패가 발생한다.
- **친구 N명 × 업데이트 주기로 비용이 곱셈으로 증가.**
  예: 친구 10명, 5초 주기, 1시간 사용 → 7,200 reads. 사용자 수가 늘면 빠르게 부담이 커진다.
- **보안 규칙 미설정 상태로 클라이언트 개발이 진행되면**, 후반에 규칙을 적용하는 순간 광범위한 권한 거부 에러가 동시 발생한다. 명세 단계에서 데이터 모델과 함께 규칙 초안을 잡아두지 않으면 회귀 비용이 크다.

---

## 7. 명세 단계에서 결론을 박아둘 항목 (참고)

본 팀이 워터폴로 진행하는 만큼, 위 유의사항들은 **구현 단계가 아닌 명세 단계에서 결정 사항으로 동결**되어야 한다. 구현 중 변경 시 영향 범위가 모듈 단위가 아니라 **앱 전체에 걸친다.**

특히 다음 항목은 **구현 진입 전 결론이 명문화되어 있지 않으면 후공정에서 재작업**이 발생할 가능성이 높다:

- 친구 공유 OFF / 강제 종료 / 권한 회수 시의 UX
- Foreground Service 알림 문구·아이콘·동작 (사용자 인지 + Play 스토어 심사 대응)
- 배터리 최적화 예외 안내 화면의 등장 시점·문구
- Activity Recognition 결과를 어떤 신호와 결합할 것인가의 판정 규칙
- Firestore 쓰기 빈도 상한과 그 도달 시 동작
- 보안 규칙 초안과 데이터 모델의 정합

## 1. 코호트·환경 차원의 선행 점검 (전 항목에 영향)

테스트 들어가기 전에 **한 번만 확인하면 76개 절반의 거짓 Fail을 막을 수 있는 항목**:

| 점검 항목 | 이유 | 조치 |
|---|---|---|
| **Firebase Console에 각 테스터 PC의 debug SHA1/SHA256 등록** | 미등록 PC는 구글 로그인이 무조건 `"구글 계정을 불러올 수 없습니다"` 토스트로 떨어짐 → 인증_04, 인증_06 이후 흐름 전체 차단 | 7명 테스터 PC에서 `keytool -list -v -keystore ~/.android/debug.keystore` 결과를 모아 일괄 등록 → google-services.json 재배포 |
| **Firestore Rules 적용 여부 (5/22 코호트 신호: 부재)** | 보안 규칙 없으면 인증 우회 가능 — 시연 위험 + 평가 감점 | 시연 전 최소한 `request.auth != null` 가드 적용 |
| **테스트용 Google 계정이 등록된 디바이스/에뮬레이터** | 계정 없는 디바이스는 인증_04에서 동일 토스트 발생 → SHA1 미등록과 동일 메시지로 진단 혼선 | 각 단말 설정 → 계정에 1개 이상 추가 |
| **에뮬레이터 이미지 = Google APIs 포함 버전** | AOSP 이미지면 Play Services 부재로 구글/푸시 모두 실패 | AVD Manager에서 "Google APIs" 또는 "Google Play" 이미지 선택 |
| **마이크/위치/알림 권한 초기화 상태** | 권한 케이스 테스트(권한_01~03, 무전_04)는 "최초 진입" 가정 → 이미 한 번 허용한 단말은 다이얼로그 미노출 | 각 케이스 전 앱 데이터 삭제 또는 신규 설치로 시작 |
| **PR 동결** | 5/27 메모: PR #194 + MapScreen 리팩토링이 회귀 2건 도입. 테스트 중 코드 변경은 fail 원인 추적을 불가능하게 함 | 6/1 09:30 이후 PR 머지 동결, fix-only로 한정 |

## 2. 도메인별 테스트 항목 + 중점 테스트 포인트

### 2-1. 인증 (8개 + 예외처리 4개)

#### 인증_001 — 로그인 화면 진입 (UI)
- **현재 결과**: Pass 5 / Fail 1(박종범) / Ready 1
- **박종범 발견 결함 2건** (반드시 cohort 회귀 점검):
  1. 가로 모드에서 로고/캐치프레이즈만 표시되고 입력창·버튼 미표시 (AOS 12·16 공통)
  2. 토스트 잔존 — 회원가입 진입 → 뒤로가기 시 직전 토스트 재표시 / 연속 취소 시 미표시
- **중점 테스트 포인트**:
  - 회전(orientation) 시 ConstraintLayout/Column scroll 처리. 가로 모드는 `android:screenOrientation="portrait"` 잠금으로 차단할지 vs 정상 지원할지 **PO 결정 필요**
  - 토스트 잔존은 **이벤트 채널의 1회성 보장 문제** — `SharedFlow(replay = 0)` 또는 `Channel`로 변경됐는지 확인. `StateFlow`로 토스트를 들고 있으면 화면 재진입 시 재발화

#### 인증_002 — 이메일 로그인 (기능 + 예외처리)
- **현재 결과**: Pass 4 / Ready 3
- **김갑석 코멘트 2건**:
  1. 키보드 노출 시 입력창이 가려짐 (스크롤 미적용)
  2. 에러 메시지가 고정 문구 — 에러 종류별 차별화 권장
- **중점 테스트 포인트**:
  - `android:windowSoftInputMode="adjustResize"` 또는 `imePadding()` 적용 여부
  - 예외처리 5종(미입력/형식오류/미인증/오인증)이 **각각 다른 토스트로 분기**되는지 확인. 현재 코드는 catch-all 패턴 의심 (SignInViewModel에서 구글 로그인이 그렇듯)
  - 이메일 미인증 케이스: 인증 메일 발송 → 미클릭 → 로그인 시도의 정확한 메시지

#### 인증_03 — 카카오 로그인 (기능 + 예외처리)
- **현재 결과**: Pass 1(장지은) / Ready 6 — **거의 미실행 상태**
- **중점 테스트 포인트**:
  - 카카오톡 앱 **설치된 단말 / 미설치 단말** 둘 다 — SDK가 카카오톡 → 카카오 계정 웹뷰로 자동 전환되는지
  - 카카오 계정 신규 사용자가 첫 로그인 시 Firestore users 문서 생성되는지 (인증_04 장지은 fail과 같은 패턴 의심)
  - 카카오 토큰 만료 후 재로그인 흐름

#### 인증_04 — 구글 소셜 로그인 (기능 + 예외처리)
- **현재 결과**: Pass 1 / **Fail 1(장지은: DB 문서 생성 안 됨)** / Ready 5
- **중점 테스트 포인트** (⭐ 최우선):
  - **장지은 fail 원인 분석**: 구글 ID Token → Firebase Auth 통과 → Firestore `users/{uid}` 문서 미생성. `signInWithCustomToken` 성공 후 user document write 로직 누락 의심. **신규 가입 vs 재로그인 분기**가 정확한지 확인 (`SetOptions.merge()` 사용 여부)
  - 본 PC 검증 결과: google-services.json에 등록된 SHA1 5개 모두 일치 안 함 → 일부 테스터 PC에서 동일 토스트 재현 예상. **SHA1 등록 작업이 인증_04 테스트의 사전조건**
  - 예외처리 4종: 현재 SignInViewModel:171-179에서 cancel만 분기되고 나머지는 모두 `"구글 계정을 불러올 수 없습니다"`로 묶임 → `NoCredentialException` / `GetCredentialUnknownException` 분기 추가하지 않으면 4종 모두 동일 토스트로 표시되어 **테스트로 검증 불가**

#### 인증_05 — 회원가입 화면 진입 (UI)

- **김갑석 코멘트 3건**: 키보드 스크롤 / 비밀번호 불일치 시 경고 미표시(버튼만 비활성) / 비밀번호 유효성 실시간 검증 미적용
- **중점 테스트 포인트**:
  - 비밀번호 정책 (자릿수, 특수문자, 대소문자)이 **인증_02(로그인)·인증_06(회원가입)·사용자정보(비번 변경)에서 일치**하는지 — 3 곳 정책 불일치는 흔한 버그

#### 인증_06 — 이메일 인증 기반 회원가입 (기능 + 예외처리)

### 5-1. 코드 정책 확인 결과

| 필드 | 하한 | 상한 | 정책 |

**리스크**:
- 닉네임 상한 0 → 1MB짜리 텍스트도 통과. Firestore 단일 필드 1MB 한계까지는 갈 수 있고, 다른 화면(친구 목록·지도 마커·DM 헤더)에서 **UI 레이아웃 파괴** + **렌더링 성능 저하** 가능

### 5-2. 비밀번호 테스트 문자열 (실효 정책 8~16자 + 허용 특수문자 14종 전수)

> **2026-06-01 16:51 박종범 발견**: Firebase 서버 정책에 **최대 16자** 상한 존재. 클라(8자 이상 무제한) ↔ 서버(16자 이하) 정책 불일치 — 17자 이상은 클라 통과 후 서버에서 `PASSWORD_DOES_NOT_MEET_REQUIREMENTS` 거부.

#### 실효 정책 (클라 ∩ Firebase 서버)

- **길이**: 8~16자 (클라 하한 8 + 서버 상한 16)

| 공백만 30자 | `[30 spaces — 본문 PowerShell로 생성]` | 30 | 30 | 같은 isBlank — 길이로 우회 시도 |

| Non-Breaking Space (U+00A0) | `닉 네임` (보통 공백처럼 보이나 U+00A0) | 4 | 11 | 일반 space와 시각 동일하나 다른 문자 — 중복 우회 |
| Tab | `닉	네임` | 4 | 10 | 입력 가능? 화면 표시 깨짐? |
| 줄바꿈 (LF) | `닉네임\n2번째줄` | 8 | 20 | ⭐ **단일 라인 enforce 안 하면 친구 목록 카드 높이 파괴** |

> Zero-width / NBSP / Tab / 줄바꿈은 직접 키보드로 입력 어려움 → PowerShell로 생성 (§ 5-6 명령 참고).

#### E. 진행 순서 권장 (박종범 단말 기준)

표시 깨짐은 **친구 목록 / 지도 마커 / DM 헤더 / 알림** 4곳 모두 확인 필요. 각 케이스 입력 후:

1. **가입 자체 통과** — 클라 차단 vs Firestore 저장
2. **내 프로필 화면 표시** — 본인 단말 표시 정상
3. **친구 등록 → 친구 목록 카드** — 카드 높이·줄바꿈·잘림
4. **지도 마커 위 닉네임** — 마커 폭 초과 잘림
5. **DM 채팅방 헤더** — 상단 바 표시
6. **알림 발신자 이름** — 시스템 푸시 + 인앱 알림 목록

총 27건 × 6 표시 위치 = 약 60분 (단말 1대).

⭐ **우선 검증 4건** (시간 부족 시 이것만이라도):
- **Zero-width Space** — 중복 닉네임 우회 보안 이슈
- **줄바꿈 (LF)** — 친구 목록 카드 레이아웃 파괴
- **ZWJ Family** — 글자 카운트 정책 결정 (1글자? 11글자?)
- **1000자 한글** — 다른 화면 OOM/렌더링 지연

### 5-6. 길이 검증·생성 PowerShell 명령

테스터가 본인 PC에서 직접 검증·생성 가능:

```powershell
# 클립보드 문자열 길이 확인
(Get-Clipboard).Length

# 임의 길이 ASCII 문자열 생성 (예: 1000자)
("a" * 1000) | Set-Clipboard

# 임의 길이 한글 문자열 생성 (예: 1000자)
("가" * 1000) | Set-Clipboard

# 비밀번호 정책 통과하는 임의 길이 (16자 이내)
("Aa1!" + ("a" * 12)) | Set-Clipboard     # 16자 (서버 상한)
("Aa1!" + ("a" * 13)) | Set-Clipboard     # 17자 (서버 거부 검증용)

# UTF-8 바이트 길이 확인 (한글이 몇 바이트로 저장되는지 — 닉네임 검증용)
[System.Text.Encoding]::UTF8.GetByteCount((Get-Clipboard))

# 닉네임 — 키보드로 입력 어려운 공백/특수문자 생성
" " * 30 | Set-Clipboard                                # 공백만 30자
"닉$([char]0x200B)네$([char]0x200B)임" | Set-Clipboard  # Zero-width Space
"닉$([char]0x00A0)네임" | Set-Clipboard                 # Non-Breaking Space
"닉`t네임" | Set-Clipboard                              # Tab
"닉네임`n2번째줄" | Set-Clipboard                       # 줄바꿈 (LF)
"닉네임`r`n2번째줄" | Set-Clipboard                     # 줄바꿈 (CRLF)

# 닉네임 — 이모지 케이스
'🔥' | Set-Clipboard                                    # 단일 이모지
'🔥👍😀🎉🚀' | Set-Clipboard                            # 이모지 5개
'👨‍👩‍👧‍👦' | Set-Clipboard                                # ZWJ Family
'🇰🇷🇺🇸🇯🇵' | Set-Clipboard                              # 국기 (regional indicator)
'👍🏻👍🏼👍🏽👍🏾👍🏿' | Set-Clipboard                       # Skin tone modifier
('🔥' * 15) | Set-Clipboard                             # 이모지 15개 (30 code units)
'안녕 🔥 닉네임' | Set-Clipboard                         # 한글+공백+이모지

# 보안 케이스
'<script>alert(1)</script>' | Set-Clipboard             # HTML/JS injection
"'; DROP TABLE users;--" | Set-Clipboard                # SQL injection 시도
```

### 5-8. 테스트 진행 순서 권장

> **닉네임**(상한 정책 없음)과 **비밀번호**(8~16자) 진행 순서가 다름. 분리해서 진행.

#### 닉네임 (정책 부재 — 무한 길이 케이스 확인)
1. **30자** — 일반 사용자 범위. 정상 통과 기대 → 통과 시 다음 단계
2. **100자** — 비정상 입력. UI 깨짐(레이아웃 overflow) 시작 지점일 가능성
3. **500자** — 화면 렌더링 한계. 입력 자체는 가능해야 하나 친구 목록·지도 마커·DM 헤더에서 표시 깨짐 예상
4. **1000자** — Firestore 통신·앱 응답성 확인. 가입 자체가 timeout 되거나, 가입 후 로그인·다른 화면 진입 시 OOM/ANR 발생 가능

각 단계 별로 확인:
- 입력 필드 자체에서 차단되는가? (`maxLength` modifier 적용?)
- 가입 버튼 누르면 어디까지 가는가? (클라 검증 → Firebase Auth → Firestore write → 다음 화면 진입)
- **다른 화면에서의 표시** — 회원가입 직후 친구 목록·지도·DM 등 닉네임이 표시되는 모든 곳 회귀 확인

#### 비밀번호 (실효 8~16자)
→ [§ 5-2 F 진행 순서 권장](#f-진행-순서-권장-박종범-단말-기준) 참고. 33건 (대조군 → 경계값 → 특수문자 전수 → 누락 → 거부 → 유니코드).

각 단계 별로 확인:
- 17자 이상 입력 시 클라단에서 사전 차단되는가? (현재 정규식 `.{8,}` 이라 차단 안 됨 — fix 필요)
- 서버 에러 `PASSWORD_DOES_NOT_MEET_REQUIREMENTS` 가 사용자에게 raw 노출되는가? (한국어 wrap 필요)
- 정책 외 특수문자 14자 입력 시 클라단에서 분명히 거부되는가? (회원가입 버튼 비활성 또는 helper text)

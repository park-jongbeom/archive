> **작성일:** 2026-05-08

## 2. 기획성 평가

### 2.1 강점

- **컨셉의 명확성과 차별화**
  현존 위치공유 앱(젠리)의 빈자리(국내 서비스 종료, 워치 미지원)를 워치 워키토키 기능으로 보강한다는 포지셔닝이 분명하다. "실시간 현장감 + 일상 기록"이라는 두 가치 축이 1차 회의에서 정리되어 있어 기능 우선순위 판단의 근거가 된다.
- **워치 활용도가 진짜 핵심 기능**
  대부분의 워치 앱이 폰 앱의 보조에 그치는 것과 달리, **PTT(Push-To-Talk)는 워치의 물리적 우위(즉시 누름·말함)**를 살린 사용 사례라 부트캠프 결과물 중 차별화 포인트가 가장 강한 영역이다.
- **역기획 전략 채택**
  1차 회의에서 "무에서 유를 창조하지 말고 젠리/카카오맵 구조를 복사·간소화" 방침을 명문화한 것은 일정·디자인 리스크를 낮추는 매우 합리적인 결정이다.

### 2.2 우려사항

- **Product Brief 미작성**
  `product_brief/Product Brief_ver 1.0` 파일이 빈 파일로 남아 있다. 컨셉·타깃 유저·성공 지표가 문서로 굳어지지 않으면 후속 디자인/기능 트레이드오프 시 의사결정 기준이 흔들린다. → **즉시 작성 필요**.
- **친구 관계 모델이 1차에서 미해결**
  1차 회의록에 "❓ 모든 사용자 vs 친구만 보이게? 친추는 어떻게?" 질문이 남아 있는데, 3차까지의 회의록에서 친구 추가 흐름·차단·프라이버시 정책에 대한 후속 결론이 보이지 않는다. PRD `친구 추가.pdf` 와의 정합성 재확인 필요.
- **위시리스트와 본 범위(In-Scope)의 경계 모호** ⚠️ *워터폴에서 가장 치명적*
  1차 위시리스트(이메일 인증·연락처 동기화·길찾기·UUID 카카오 공유·히스토리 등)가 **본 구현 범위인지 추후 확장인지 표시가 없다.** 워터폴에서는 구현 단계 진입 후 "이거 해야 해? 빼야 해?" 논의가 발생하면 명세 회귀(re-spec) 비용이 발생한다. **명세 단계 종료 직전에 위시리스트 전 항목을 In-Scope / Out-of-Scope 두 칸으로 강제 분류**하고 동결할 것.

## 3. 작업 가능성(기술 실현성) 평가

### 3.1 위치 공유 — **실현 가능 (★★★★☆)**

3차 회의에서 다룬 전략이 표준에 부합한다.

- ✅ Firestore 기반 + 5초 throttling + 필요 시 RTDB 하이브리드
- ✅ 클라이언트 보간(Interpolation) 적용 의사결정
- ✅ Activity Recognition + FusedLocationProvider 조합 (배터리 최적화 표준)
- ✅ Foreground Service + `FOREGROUND_SERVICE_LOCATION` (Android 14+ 대응)

> **체크 포인트:** 3차에서 `minSdk` 를 "34~34"로 변경한다는 메모가 있는데, **minSdk 34는 시장 커버리지를 크게 좁힌다**(2026년 기준 다수 단말이 미충족). 백그라운드 서비스 타입 선언만 필요하다면 `minSdk 24~26` 유지 + `targetSdk 34/35`로 가는 것이 일반적이다. 강사 피드백 재확인 권장.

### 3.2 워키토키 (Wear OS) — **가장 큰 리스크 (★★☆☆☆)**

- 3차 회의 권고대로 **"워치 → 폰 → Firebase"** 경유 아키텍처가 안정적이다 (배터리·연결성 측면 정답).
- 다만 다음 4가지가 동시에 처음 접하는 영역이라 1주 일정에 매우 빡빡하다:
  1. Wear OS 모듈 빌드/서명 일치
  2. `DataLayer API` (DataClient/MessageClient/**ChannelClient**) 분기 구현
  3. 오디오 코덱(Opus/AAC) 인코딩 + 청크 전송
  4. Firebase Storage 업로드 + 메타데이터 RTDB 트리거

> **권고 (워터폴 전제):**
> 워터폴에서는 "일단 구현하고 안 되면 줄인다"가 불가능하다. **명세 단계에서 워키토키 스펙을 한 가지로 확정**해야 한다. 두 안 중 택일:
> - **(A) 보수안)** 녹음 → 폰 경유 업로드 → 상대방 재생 = 단방향 음성 메시지. 3차 회의 합의("처음 구현하신다면 폰 경유 먼저")와 일치하며 일정·구현 리스크 모두 낮음. **데모 성공 확률 우선이라면 A.**
> - **(B) 도전안)** ChannelClient 기반 실시간 PTT 스트리밍. 차별화 가치는 크지만 명세 단계에서 **PoC(2~3일)를 반드시 선행**해 실제 구현 가능 여부를 확정한 뒤 본 명세에 반영.
>
> **PoC 없이 B안을 명세에 못 박으면 워터폴 후공정에서 일정 붕괴 위험이 가장 크다.**

### 3.3 DM (실시간 메시지) — **실현 가능 (★★★★☆)**

PRD에 별도로 분리되어 있는데, Firestore subcollection + Snapshot listener로 충분히 구현 가능. 다만 워키토키와 DM이 **동시에 백그라운드 리스너 + 푸시까지 요구**하므로 자원 경합 설계를 한 번 그려두는 것이 좋다.

### 3.4 일정·인력 리스크

- 4인 / Android+Wear OS 동시 / 백엔드는 Firebase로 흡수 → **인력 배치는 합리적**.
- 그러나 워치 빌드·실기기 디버깅은 1인 풀타임이 필요하고, 위치 Foreground Service도 1인 풀타임이 필요하다. **나머지 2인이 Auth/친구/DM/UI를 모두 커버**하게 되어 빠듯하다.
- 3차 TODO의 "기능명세서/화면명세서 완료, DB 설계 마치기, 깃 규칙, 프로젝트 기초 설정"이 한 날에 묶여 있다. 워터폴에서는 이 4개가 **모두 완료되어야 구현 단계 진입**이 가능하므로, **명세 단계에 충분한 시간을 더 할당**하는 것이 옳다. 부족한 명세로 구현 진입 → 도중 명세 변경 → 코드 폐기는 워터폴에서 가장 큰 손실이다. 화면명세는 1차 회의에서 제시된 v0.dev/uizard 결과를 명세 산출물로 채택하면 시간을 단축할 수 있다.

## 4. 핵심 리스크 Top 5 (워터폴 관점)

워터폴은 후공정에서 명세 변경 비용이 비대칭적으로 크다. 따라서 **"명세 단계에서 미리 잠가야 할 항목"** 위주로 리스크를 재배열한다.

| # | 리스크 | 명세 단계에서 잠가야 할 결정 | 잠그지 못했을 때 |
|---|---|---|---|
| 1 | 워치 워키토키 스펙 불확정 | 단방향 음성메시지(A) vs 실시간 PTT(B) 택일. B안 시 **PoC 선행 필수** | 구현 중 일정 붕괴 → 데모 실패 |
| 2 | Product Brief / 친구 정책 미문서화 | Brief v1.0 작성, 친구 양방향 여부·차단·공유 단위 확정 | 친구/위치/DM 모듈이 동시에 영향받아 광범위 재작업 |
| 3 | In-Scope / Out-of-Scope 미분리 | 1차 위시리스트 전 항목을 두 칸으로 강제 분류, **동결** | 구현 중 "이거 해야 해?" 논쟁 → 명세 회귀 |
| 4 | minSdk 34 결정 | 강사 피드백 반영해 `minSdk` 명확화 (24~26 권장) | 테스트 단말 확보 실패 → 데모 환경 부재 |
| 5 | Firestore 보안 규칙·DB 스키마 | 3차에서 "배포 전에 하기" 결정했으나, **데이터 모델은 명세 단계에서 확정** 필요 | 후반 변경 시 마이그레이션·코드 재작성 |

> 위 5개는 **모두 구현 단계 진입 전(=명세 단계에서) 결정 가능한 항목**이다. 워터폴의 강점인 "선행 결정"을 활용해 잠그면 후공정 리스크가 크게 줄어든다.

## 5. 잘된 점 (긍정 시그널)

1. **3차 회의록의 기술 리서치 깊이가 상당하다** — Snapshot 비용 분석, 스냅샷 vs 폴링 충돌, Doze 모드, WakeLock, WorkManager vs Foreground Service 비교까지 실무 수준의 검토가 들어가 있어 **팀의 기술 이해도가 높음**을 보여준다.
2. **PRD가 기능별로 11개 분리**되어 있어 작업 분배 단위가 자연스럽다.
3. **로고/네이밍(BBip)**까지 시각화 안이 나와 있어 데모 발표 시 완성도 신호로 작용 가능.
4. **이슈 기반 회의 운영** — 의견 충돌이 아닌 "이슈사항 → 결론" 구조로 진행되어 의사결정 트래킹이 용이하다.

## 6. 권고사항 (워터폴 단계별 Action Items)

> 워터폴 단계: **요구사항/명세 → 설계 → 구현 → 통합/테스트 → 배포**
> 현재 위치는 **요구사항/명세 후반**으로 보인다.

### Phase 1. 요구사항·명세 단계 (구현 진입 전 반드시 잠금)
- [ ] **Product Brief v1.0 작성 완료** (현재 빈 파일) — 컨셉/타깃/성공 지표 1쪽
- [ ] **In-Scope / Out-of-Scope 분류표 작성** — 1차 위시리스트 9개 항목 전부 두 칸으로 강제 분류 후 **동결 서명**
- [ ] **친구 관계 정책 확정** — 양방향 친추 / 차단 / 공유 단위 (전체·친구별)
- [ ] **워키토키 스펙 택일** — A(단방향 음성메시지) / B(실시간 PTT, PoC 선행). 본 리뷰는 **A안 권고**
- [ ] **minSdk 결정 재확인** (강사 피드백)
- [ ] **DB 스키마 표 1장 + Firestore 보안 규칙 초안** (1차 회의 가이드 따라 스프레드시트로)

### Phase 2. 설계 단계
- [ ] **화면명세** — v0.dev/uizard 결과를 산출물로 채택, 컴포넌트 단위 분해
- [ ] **모듈/패키지 구조** — `app(Phone)` / `wear(Watch)` / `service(DataLayer)` 의 책임 경계 문서화
- [ ] **부하/비용 시나리오 사전 검증** — 친구 10명 × 5초 업데이트 1시간 = 7,200 reads. 데모 규모 시뮬레이션
- [ ] **Wear OS 실기기 빌드/페어링 PoC** — 에뮬레이터 PTT 검증 불가, **설계 단계에서 한 번은 실기기 통신 성공해두기**

### Phase 3. 구현 단계 진입 조건
- 위 Phase 1·2 산출물이 모두 동결되었을 때만 구현 진입.
- 구현 중 명세 변경이 발생하면 **명세 단계로 회귀 → 영향받는 모든 모듈 재검토** 후 진행 (워터폴 원칙 준수).

### Phase 4. 통합·테스트 / 배포
- 보안 규칙은 **배포 직전이 아닌 통합 테스트 단계에서 적용**해 권한 이슈를 미리 노출시킬 것.

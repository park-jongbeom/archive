## 0. 이 가이드의 한 줄 전제

**신입 발표에서 면접관이 사는 것은 "완성된 결과물"이 아니라 "이 사람이 입사 후 빠르게 성장하고 팀에 녹아들까"라는 예측이다.**

그래서 의뢰인(채용 실무자)이 강조한 3가지 — **트러블슈팅 · 협업 규칙 · 회고와 개선** — 이 정확히 채용 신호의 정중앙이다. 각각이 증명하는 것:

| 영역 | 면접관에게 증명하는 것 |
|---|---|
| **트러블슈팅** | 학습 능력 · 문제를 쪼개는 사고력 · 기술 의사결정 |
| **협업 규칙과 준수** | 팀 적합성 · 책임감 · 프로세스 감각 |
| **회고와 개선** | 자기개선 루프 · 메타인지 · 성장 의지 |

**가장 흔한 실패는 "정했다 / 했다 / 고쳤다"로 문장을 닫는 것이다.** 모든 항목은 **증거(스크린샷·로그·PR 링크·수치)와 before/after 변화**로 닫아야 신호가 산다. 이 가이드 전체를 관통하는 두 단어는 **증거화**와 **정량화**다.

## 1. 발표 전체 슬라이드 구조 (13분 기준)

YC 데모데이 원칙(슬라이드당 1메시지, 라이브 데모 금지·녹화 권장, 핵심 3~4개 명시 각인)을 신입 발표에 맞게 변형. **3대 영역(⭐)에 전체 시간의 절반 이상**을 배분하는 것이 차별화의 핵심이다.

| # | 슬라이드 | 한 줄 내용 | 시간 |
|---|---|---|---|
| 1 | 타이틀 / 팀 | 프로젝트명 + 한 줄 정의 + 팀원 각자 역할 1줄 | 0:30 |
| 2 | 문제 · 해결 (Hook) | "누구의 어떤 문제를 푸는가" — 군더더기 없이 | 1:00 |
| 3 | 데모 | **녹화 영상 60~90초** (라이브 금지), 핵심 플로우만 | 1:30 |
| 4 | 아키텍처 / 기술스택 | 다이어그램 1장 + **"왜 이 기술을 골랐나"** 근거 | 1:30 |
| 5 | **트러블슈팅 ①②** ⭐ | 대표 2건: 문제→가설→검증→해결→**before/after 수치** | 3:00 |
| 6 | **협업 Ground Rules** ⭐ | 규칙 5~6개 + **준수율 / PR·리뷰 수치 증거** | 2:00 |
| 7 | **회고 (KPT)** ⭐ | "안 지켜진 것 인정 → 개선 액션 → 결과 변화" 1~2건 | 2:00 |
| 8 | 성과 지표 | 크래시율 / 커버리지 / 사이클타임 등 정량 요약 | 1:00 |
| 9 | 마무리 · 배운 점 | 각인할 핵심 3~4개 명시 | 0:30 |

**설득의 흐름**: 문제 → 데모(된다는 증명) → 어떻게 만들었나(기술) → 고생한 것(트러블슈팅) → 어떻게 함께 일했나(협업) → 무엇을 배웠나(회고). 결과를 먼저 보여주고, 과정의 깊이로 신뢰를 쌓는다.

> 발표 시간이 7~8분이면 슬라이드 5·6·7만 남기고 1·8을 압축. 3대 영역은 절대 빼지 않는다.

## 2. 핵심 3대 영역 작성법

### 2-1. 트러블슈팅 — "해결책"이 아니라 "삽질의 궤적"이 자산

한국 채용담당자들이 직접 권하는 구조는 단순 `문제→해결→결과`가 아니라 **`문제 → 시도한 방법들(가설·검증) → 비교 → 알게 된 점`**이다. 틀린 가설(삽질)조차 사고 과정을 증명하기 때문.

**5단계 템플릿**
1. **문제 정의** — 무엇이, 어떤 영향을. **반드시 숫자/증상** (예: "스크롤 시 평균 38fps, ANR 2건")
2. **가설 / 시도한 방법들** ← *핵심.* "내가 세운 가설들과 검증 과정". 틀린 시도도 보여준다.
3. **원인 분석** — 로그·프로파일링 근거와 함께 진짜 원인
4. **해결** — 번호 매긴 단계 + 코드 스니펫 + **왜 이 해결책을 골랐나(트레이드오프)**
5. **결과 · 학습** — **before → after 정량 수치** + 재발 방지책

**좋은 예 / 나쁜 예**

| 단계 | ❌ 나쁜 예 | ✅ 좋은 예 |
|---|---|---|
| 문제 | "느렸다" | "100개+ 아이템 스크롤 시 프레임 드랍, 평균 38fps" |
| 과정 | (생략, 바로 해결로 점프) | "①DiffUtil 가설→부분개선 ②이미지가 메인스레드 점유 발견→Glide 전환 ③측정 비교" |
| 결과 | "빨라졌다" | "38→58fps, jank 90%↓, ANR 0건" |
| 학습 | (없음) | "UI 스레드 블로킹이 ANR 주원인임을 체득, 이후 StrictMode 상시 적용" |

> ⚠️ "이것저것 해봤더니 됐다"는 오히려 **감점**. 무작정 삽질이 아니라 **가설→검증 사이클**이었음을 보여야 가치가 생긴다.

### 2-2. 협업 규칙 — "정했다"가 아니라 "지켰다/안 지켰다"를 증거로

규칙 나열은 누구나 한다. **차별화는 준수율 수치와 동기(왜 만들었나)에 있다.**

**규칙 카테고리** (5~6개면 충분): ① 브랜치 전략 ② 커밋 컨벤션 ③ PR 리뷰 룰 ④ 이슈/PR 템플릿 ⑤ 회의·스탠드업 규칙 ⑥ 보안·위생(시크릿 관리)

**증거화 변환표**

| 약한 서술(나열) | 강한 서술(증거+준수율) |
|---|---|
| "커밋 컨벤션 사용" | "커밋 312건 중 287건(92%) 컨벤션 준수 — git log 스크린샷" |
| "코드 리뷰 진행" | "PR 64건, 평균 리뷰어 2명, 머지까지 평균 6h, 리뷰 코멘트 210건" |
| "매일 스탠드업" | "20일 중 17일(85%), 미진행 3일은 시험기간 → 비동기 노션 스탠드업으로 전환" |

**3가지 원칙**
- **준수율을 %로**: 규칙이 장식이 아니라 실재했음을 증명.
- **안 지켜진 것을 먼저 인정**: "100% 지켰다"는 오히려 의심받는다. (→ 2-3 회고로 연결)
- **규칙의 동기를 붙인다**: "반복 질문을 개인 탓이 아닌 시스템 문제로 보고 → 변수명·주석 규칙을 만들었다"처럼 문제의식을 보이면 성숙해 보인다.

> 적은 규칙을 확실히 지킨 증거 > 많은 규칙 나열. 5~6개 규칙, 각각 준수 증거 1개씩이 이상적.

### 2-3. 회고 · 개선 — "문제 인지 → 개선 액션 → 결과 변화"의 닫힌 루프

발표 슬라이드에는 **KPT(Keep/Problem/Try)를 표 1장**으로 압축(한국 면접관에게 가장 친숙). 단순 나열("Keep: 소통 잘함, Problem: 일정 빠듯")은 죽은 회고다. 반드시 **클로즈드 루프**로:

**발표용 1줄 공식**: `[문제 인지] → [구체적 개선 액션] → [측정된 결과 변화]`
- 예: "초기 PR이 너무 커서 리뷰가 형식적(문제) → 200줄 이하로 쪼개는 룰 도입(액션) → 리뷰 시간 6h→1.5h, 머지 후 버그 4건→1건(결과)"

**추상적 Try 금지**: "공유를 잘하자" ❌ → "회의록 자동 발송" ✅

**실패·갈등을 성숙하게 다루는 톤 (blameless 원칙)**: 개인이 아니라 **시스템·프로세스로 프레이밍**.

| ❌ 비난 | ✅ 시스템 |
|---|---|
| "A가 리뷰 없이 머지했다" | "리뷰 없이 머지가 가능한 프로세스였다 → 브랜치 보호 규칙 추가" |
| "팀원이 마감을 놓쳤다" | "산정이 통합 복잡도를 반영 못 했다 → 버퍼 산정 도입" |

> "갈등 없이 화목했다"는 면접관이 **가장 싫어하는 거짓 신호**다. 갈등은 "있었다 + 어떻게 풀었다"가 정답.

## 3. 가장 강력한 한 방 — 단일 내러티브

발표의 중심축으로 삼을 단 하나의 사례는 이것이다:

> **"잘 안 지켜진 규칙을 인정 → 회고로 개선 → 수치로 변화 증명"**

이 내러티브 하나가 **트러블슈팅·협업·회고·성장 의지를 동시에** 증명한다. 우리 코호트 3팀 모두 이 축으로 쓸 실제 사례가 있다(아래 6장). 슬라이드 6·7을 이 한 사례로 꿰면 발표가 기억에 남는다.

## 4. 안티패턴 체크리스트 (하지 말 것)

**내용 차원**
- [ ] 기능 나열형 ("이런 기능 만들었어요" 카탈로그, 고민 0)
- [ ] 결과만 자랑 (before/after 수치 없음)
- [ ] 기술 스택 로고만 나열 ("왜 골랐나"가 없음 — 가장 흔한 감점)
- [ ] 개인 기여 불명확 (팀 성과 속 "내가 한 일"이 안 보임 — 주어 "나는" 누락)
- [ ] "갈등 없이 화목했다" (problem이 없는 회고)
- [ ] 트러블슈팅에 가설·검증 없이 "해보니 됨" 또는 고생 없는 "바로 고침"
- [ ] 추상적 회고 ("소통 잘하자" 같은 측정 불가능한 Try)

**발표 차원 (YC)**
- [ ] 라이브 데모 강행 (현장에서 거의 실패 → 녹화로)
- [ ] 슬라이드 글자 폭탄 (청중이 읽느라 안 들음)
- [ ] 슬라이드만 쳐다보기 / 빠른 말하기 / 멈춤 없음
- [ ] 과장 ("182x 빨라졌다" 같은 검증 불가 주장 — 작은 거짓도 치명적)

## 5. 정량화 지표 메뉴 (실제로 측정 가능한 숫자)

부트캠프 규모에서도 GitHub · Play Console로 대부분 산출 가능.

**A. 협업 지표**
- PR 수 / 인당 PR 수, PR당 평균 리뷰어 수
- PR 사이클 타임(생성→머지 중앙값)
- 코드 리뷰 코멘트 총수 / PR당 코멘트 수
- 커밋 컨벤션 준수율(%), 브랜치 전략 준수율
- 회의/스탠드업 준수율(%), 회고 실행 횟수
- Decision Log 누적 건수

**B. 트러블슈팅·성능 (before→after)**
- 응답/로딩 시간 (4.2s→1.1s, -74%)
- 프레임레이트 / jank (38→58fps)
- API 호출 수 / 네트워크 트래픽 감소율
- 빌드 시간, 앱 용량(AAB, ProGuard 전후)
- 버그 해결 시간, 재현율

**C. 품질·안정성 (안드로이드 — Play Console / Android vitals)**
- 크래시율(%), 사용자 인식 ANR 발생률
- 테스트 커버리지(%), Lint/정적분석 경고 수 감소
- 앱 시작 시간(cold start), 메모리 누수 건수

> ⚠️ DORA 4대 지표는 "개념을 의식하고 파이프라인을 구성했다" 정도로만. "우리도 elite다"는 과장으로 비친다.

> **워터폴 팀(3팀) 주의**: 협업 규칙을 스프린트가 아니라 "단계 종료 기준·명세 완성도·범위 잠금" 프레임으로 서술. "MVP 다이어트/빠른 반복"으로 포장하지 말 것.

---

## 7. 발표 전달 팁 (YC)

- 천천히 말하고 중간중간 멈춘다.
- 슬라이드가 아니라 청중을 본다.
- 가장 잘 말하는 사람이 발표한다(팀장일 필요 없음).
- 슬라이드 글자와 말이 충돌하지 않게(슬라이드는 키워드, 말로 살을 붙임).
- 데모는 녹화 영상. 핵심 플로우 하나만 60~90초.

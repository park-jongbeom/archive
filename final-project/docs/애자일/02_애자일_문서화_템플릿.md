# 1. GitHub — Issue 템플릿 (핵심)

> **메인 가이드:** `01_애자일_팀프로젝트_가이드.md`  
> 이 문서는 “운영 규칙”이 아니라, **Issue/PR/Notion 문서를 어떻게 쓰는지(복붙 템플릿)**만 제공합니다.

> 참고: 이 템플릿 하나가 “백로그 + 기능명세” 역할

### Issue 작성 규칙(Ready 관련)
- 작은 Flow(예: 로그아웃)는 **Flow 1개 = Issue 1개 = PR 1개**로 끝내도 된다. (기준은 “PR이 하루 내 리뷰 가능한가”)
- **UI 변경/신규 화면/컴포넌트**가 포함된 Issue는 `Related`에 **Design(Figma) 링크**가 없으면 Ready로 올리지 않는다.
- 링크는 가능하면 Figma 파일 전체가 아니라 **해당 Frame 바로가기 링크**를 사용한다.

> 용어(빠른 정의)
> - **Flow(User Flow)**: 사용자 목표 1개를 달성하는 시작→끝 흐름(설명용).
> - **Story/Issue**: Flow를 구현하기 위한 개발 작업 단위(실행 기준(SSOT)).
> - **완료 기준(AC)**: Done 판정 체크리스트. **완료 기준(AC) 없으면 Ready 금지**.
> - **Ready**: 이번 주 착수 가능 상태(완료 기준(AC) + Flow 링크 + 주차(Iteration, Week) 지정).
```markdown
# [Feature] 로그인 기능

## User Story
사용자는 로그인할 수 있다.

---

## 완료 기준(AC) (Acceptance Criteria)
- [ ] 이메일 형식 검증
- [ ] 비밀번호 8자 이상
- [ ] 로그인 성공 시 홈 화면 이동
- [ ] 로그인 실패 시 에러 메시지 표시
- [ ] 5회 실패 시 계정 잠금 (30분)

---

## Flow (링크)
- Flow 문서 링크(예: Notion 또는 `example/02_flow/flow_login.md`)

---

## (선택) Details (필요 시만)
> 원칙: **결정된 정책 2~5줄만** 적는다. (긴 화면명세/기획문서 금지)
>
> 예시(참고용):
> - 인증 방식: (예: 토큰 기반 / Firebase Auth 등)
> - 세션/만료 정책: (예: 앱 재실행 시 유지 여부)

---

## 예외(Edge Cases)
- 네트워크 오류
- 서버 500 에러
- 권한 거부
- 데이터 없음(Empty)
- 중복 제출(연타)

> 권장: Edge Cases는 **3~5개 이상**을 최소로 둔다. (메인 가이드 `01_...`의 Flow 품질 기준과 동일)

---

## Related
- PR: #
- API: (있으면 링크)
- Design(Figma): (UI 포함 시 필수, Frame 링크 권장)
  - 와이어프레임 체크(최소):
    - 화면에 있어야 할 요소(박스 수준)가 합의됐는가?
    - 로딩(Loading)/실패(Error)/데이터 없음(Empty, 필요 시)의 위치가 표시됐는가?
    - Empty/에러의 핵심 문구와 CTA(다음 행동)가 정해졌는가?
    - 화면 전환/뒤로가기/종료 조건이 1줄로라도 합의됐는가?

---

## Labels
- type: feature
- priority: high
```

# 2. GitHub — PR 템플릿

> 용어(판정용)
> - **Done**: PR Merge + 완료 기준(AC) 충족.
> - **실행 기준(SSOT)**: 실행 기준 1곳(보통 Issue). PR에서 `Closes #123`로 Issue↔PR을 명시적으로 연결.
```markdown
## Summary
- 로그인 기능 구현

---

## Related Issue
- Closes #123

---

## What’s Done
- 로그인 API 연동
- 에러 처리
- UI 연결

---

## How to Test
1. 이메일/비밀번호 입력
2. 로그인 버튼 클릭
3. 성공/실패 확인

---

## Notes
- 남은 작업/리스크 (예: 실패 케이스 추가, 재시도 경로 보강 등)
```

→ 홈 화면 이동

### 실패

- Loading

## 4-1. (권장) Notion 목록형(Flow DB) 템플릿(복붙용)

Flow는 Notion에서 “문서 여러 장”보다 **목록(DB) + 상세 페이지**로 운영하는 게 관리가 쉽습니다.
단, **Design(Figma) 링크는 Flow가 아니라 UI 포함 Issue의 `Related`에 둡니다.** (Flow는 흐름/상태 합의 문서)

### Flow DB(목록) 추천 필드(6~8개)
- Flow ID (예: `FLOW-LOGIN`)
- Flow 이름
- 목표(1줄)
- Owner(담당자 1명)
- 상태 (예: 초안 / 확정 / 변경됨)
- 주차(Iteration, Week) (예: Week2)
- 데모 포함 (예/아니오)
- 관련 링크 (Flow 페이지/이슈 링크)

### 목록형 시각화(예시)

| Flow ID | Flow 이름 | 목표(1줄) | Owner | 상태 | 주차(Iteration, Week) | 데모 포함 | 관련 링크 |
|---|---|---|---|---|---|---|---|
| FLOW-LOGIN | 로그인 | 로그인하고 홈으로 간다 | 김OO | 확정 | Week2 | 예 | `example/02_flow/flow_login.md` |
| FLOW-LOGOUT | 로그아웃 | 로그아웃하고 로그인 화면으로 간다 | 이OO | 확정 | Week2 | 예 | `example/02_flow/flow_logout.md` |
| FLOW-HOME | 홈 목록 | 목록을 보고 항목을 선택한다 | 박OO | 초안 | Week3 | 아니오 | `example/02_flow/flow_home_list.md` |

### Flow 상세 페이지 템플릿(최소 구성)

```markdown
# User Flow — {Flow 이름} ({Flow ID})

## 목표
- (사용자 목표 1줄)

## 시작 조건
- (로그인 상태/권한 상태 등)

## 주요 단계
1. 사용자 행동
2. 시스템 반응(로딩(Loading) 등)
3. 성공/실패 분기

## 성공
- (어디까지 도달)

## 실패
- (무엇이 보이고, 재시도는 가능한가)

## 상태(필수)
- 로딩(Loading)
- 실패(Error)
- 데이터 없음(Empty, 필요 시)

## 예외(Edge Cases)
- 3~5개 이상

## 연결
- GitHub Issue(실행 기준(SSOT))
- 데모 시나리오(Demo Scenario)
- 디자인(Figma, 필요 시)
```

# 6. Notion — Decision Log 템플릿

## 6-1. (권장) 목록(DB)+상세로 운영하는 예시

### Decision DB(목록) 추천 필드(복붙용)
- Decision ID (예: `DEC-001`)
- 제목
- 날짜
- 상태(예: 제안 / 확정 / 폐기)
- Owner(담당자 1명)
- 영향 범위(예: FLOW-LOGIN, auth, home 등)
- 관련 링크 (Issue/PR)

### 상세 페이지 템플릿(최소 구성)
- 문제(무엇이 문제였나?)
- 선택지(2~3개)
- 결정(무엇으로 정했나?)
- 이유/영향(왜 그렇게 했고, 무엇이 바뀌나?)

> 팁: 범위 변경/규칙 예외/기술 선택처럼 “나중에 다시 물어볼 것”만 Decision으로 남깁니다.

# 9. 최종 구조 요약

## GitHub

* Issue = **실행 + 명세**
* PR = **구현 + 검증**

---

## Notion

* Brief = 왜
* Flow = 어떻게 흐르는지
* Demo = 어떻게 보여줄지
* Decision = 왜 그렇게 했는지

---

# 부록. Notion 문서 운영 방식(짧게)

- 기본은 **목록(DB)+상세 페이지**로 운영하는 것을 권장합니다. (주차/담당자/상태/발표 포함 필터가 편함)
- 단, Flow가 1~2개이고 문서가 부담이면 **단일 문서(1페이지)**로 시작해도 됩니다.
- 2주차부터 “이번 주 데모/결정/해야 할 일”을 빠르게 모아봐야 한다면, DB가 더 편합니다.
- 원칙은 동일합니다: Notion은 “설명/정리용”, 개발 실행 기준(SSOT)은 GitHub Issue입니다.

# 10. 가장 중요한 한 줄

> **“개발은 GitHub만 보고 가능해야 한다”**

---

이 템플릿 그대로 쓰면:

* 과도한 문서 방지
* 명세 부족 방지
* 협업 꼬임 방지

---

원하면 다음 단계로
- GitHub Project 보드 세팅 (컬럼/자동화)
- Issue 라벨 구조 설계
- 팀원 역할별 사용 가이드

여기까지 붙이면 그냥 바로 팀 운영 가능하다.

# 0. 기본 전제 (모든 역할 공통)

> **메인 가이드:** `01_애자일_팀프로젝트_가이드.md`  
> 이 문서는 팀 내 역할(PO/Dev/Reviewer/QA)을 **현실적으로 어떻게 굴릴지**를 역할별 체크리스트로 정리했습니다.

* **GitHub Issue = 실행 기준(SSOT)**
* **PR 없으면 작업 안 한 것**
* **완료 기준(AC) 충족 = Done**
* 상태 변경은 “행동으로 자동 반영” (수동 최소화)

> 용어(빠른 정의)
> - **실행 기준(SSOT)**: 진짜 기준 1곳. 여기서는 GitHub **Issue**.
> - **완료 기준(AC)**: Done 판정 체크리스트(관찰 가능/예-아니오).
> - **Done**: PR Merge + 완료 기준(AC) 충족.

## 팀장/부팀장 운영 매핑(PO 전담 없음)

> 팀장/부팀장 체계가 있는 팀은 아래처럼 **PO 역할을 겸임**하면 된다.

- **팀장(스코프/Ready 책임)**: 이번 주에 “무엇을 할지”를 정한다.
  - 이번 주 목표 1줄 확정
  - Ready 확정(완료 기준(AC) 있는 이슈만)
  - 범위 변경 승인(필요 시 Decision Log 1줄)
- **부팀장(품질/데모 책임)**: “데모가 실패 없이 돌아가게” 품질을 챙긴다.
  - 리뷰 응답 시간(권장 24h) 관리
  - 데모 시나리오/회귀 체크
  - Week 6 리허설 주도
- (UI 포함 시) **와이어프레임 합의/변경 승인**: 이번 주 Demo에 포함되는 화면은 **Frame 링크 1개로 기준을 남긴다**.
  - 변경이 생기면 Issue 코멘트 2~5줄로 요약한다.

> 용어
> - **Ready**: 이번 주 착수 가능 상태(완료 기준(AC) + Flow 링크 + 주차(Iteration, Week) 지정).
> - **리뷰 응답 시간**: PR 리뷰 응답 시간 기준(예: 24h 내 1차 피드백).
> - **데모 시나리오**: 실제로 눌러 보여줄 실행 순서(점검 기준).

## (중요) 4~5명 팀 업무 할당 규칙(최소)

역할을 복잡하게 나누기보다, 아래 6가지만 고정하면 “일이 떠돌지” 않습니다.

- **Issue 1개 = Owner 1명(필수)**: 해당 Issue는 Owner가 끝까지 책임진다(완료 기준(AC) 충족, PR, 수정 대응).
- **In Progress는 1인당 1~2개 제한**: 동시에 너무 많이 잡지 않는다.
- **Ready는 팀장(또는 합의된 1명)만 확정**: Ready에 올라간 것만 이번 주에 진행한다.
- **PR은 반드시 Issue를 닫는다**: PR 본문에 `Closes #123`로 연결한다.
- **리뷰 응답 시간(권장 24h)**: PR이 올라오면 24시간 내 1차 피드백(승인/수정 요청/질문)을 남긴다.
- **UI가 포함되면 Design(Figma) 링크가 있는 Issue만 착수**: 화면 기준이 바뀌면 Issue 코멘트로 2~5줄 요약한다.

## 4인/5인 추천 운영표(예시)

> 원칙: “역할”보다 “책임”을 나눕니다. 한 사람이 여러 역할을 겸임해도 됩니다.

### 4인 팀
- **팀장**: Ready 확정 + 범위 변경 승인(필요 시 Decision)
- **부팀장**: 데모 시나리오/회귀 체크 + 리뷰 응답 시간(권장 24h) 관리
- **Dev A**: Issue Owner(기능 1) + 교차 리뷰
- **Dev B**: Issue Owner(기능 2) + 교차 리뷰

### 5인 팀
- **팀장**: Ready 확정 + 범위 변경 승인(필요 시 Decision)
- **부팀장**: 데모 시나리오/회귀 체크 + 리뷰 응답 시간(권장 24h) 관리
- **Dev A**: Issue Owner(기능 1) + 교차 리뷰
- **Dev B**: Issue Owner(기능 2) + 교차 리뷰
- **Dev C**: Issue Owner(기능 3) + 교차 리뷰

# 1. PO / 기획 (또는 팀 리드)

## 역할

* “무엇을 왜 만드는지” 책임

---

## 해야 할 일

### Week 1

* Notion Product Brief 작성
* Flow 정의
* Demo Scenario 초안 작성

---

### 매 스프린트 시작

* Backlog 우선순위 정렬
* Ready 상태로 올릴 Issue 선정

---

### 개발 중

* Flow 변경 시 Notion 업데이트
* 모호한 요구사항 명확화

---

### Week 6

* Demo Scenario 확정
* 발표 스토리 구성

---

## 하지 말 것

* GitHub Issue에 상세 명세 작성 금지
* 개발 방식 간섭 금지

---

## Done 기준
* Flow 1~2개가 확정되고(목표/시작/끝/실패 포함)
* 이번 주 Ready가 확정되며(완료 기준(AC) + 주차(Iteration, Week) + Flow 링크)
* Demo Scenario 초안이 있다

---

# 2. 개발자 (Dev)

## 역할

* “Issue를 실제 동작으로 만드는 사람”

---

## 해야 할 일

### 작업 시작 전

* Issue 확인
* 완료 기준(AC) 이해
* 부족하면 질문

---

### 작업 시작

* Assignee 지정 → 자동으로 In Progress
* 브랜치 생성

---

### 개발 중

* 완료 기준(AC) 기준으로 구현
* 필요한 경우 Issue에 코멘트로 보완

---

### 완료 시

* PR 생성
* 반드시 Issue 연결:

```text
Closes #123
```

---

### 리뷰 대응

* 피드백 반영
* 수정 후 다시 요청

---

## 하지 말 것

* 완료 기준(AC) 없는 상태에서 개발 금지
* Issue 없이 작업 금지
* PR 없이 완료 처리 금지

---

## Done 기준

* PR Merge + 완료 기준(AC) 모두 충족

---

# 3. 리뷰어 (Reviewer)

## 역할

* 코드 품질 + 요구사항 충족 검증

---

## 해야 할 일

### PR 확인 시

#### ① 요구사항 체크

* Issue 완료 기준(AC) 충족 여부

#### ② 코드 리뷰

* 가독성
* 구조
* 중복

#### ③ 동작 확인 (가능하면)

* 실제 테스트

---

### 리뷰 결과

* 코멘트 남김
* Approve / Request changes

---

## 하지 말 것

* 기능 요구사항 추가 금지 (PO 역할)
* 과도한 리팩터링 요구 금지

---

## Done 기준

* Issue의 완료 기준(AC)을 기준으로 예/아니오 판단이 끝났다
* PR의 `How to Test`를 최소 1회 따라 해보고 동작이 확인됐다(가능하면)
* 리스크/남은 작업이 있으면 PR Notes에 1~3줄로 남겼다

# 5. 전체 협업 흐름 (역할 연결)

```text
PO → Flow / 방향 정의
→ Issue 생성 (GitHub)

Dev → Issue 구현
→ PR 생성

Reviewer → 리뷰

QA → 시나리오 검증

→ Done
```

# 10. 한 줄 정리

> Dev는 동작을 만든다
> Reviewer는 품질을 보장한다
